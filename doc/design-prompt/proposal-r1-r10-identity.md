### 1.2 问题（product-vision §1 X 光诊断，需求源，非本卡自造）

vision §1 line 20 把首句读出声后定性两处病灶：**①「新闻信息流」= 仍自认信息产品**（宿命是「更多源、更快、更全」的军备竞赛，零服务器独立 App 在那条赛道无位置）；**②「对标 Ambient News Globe 形态」= 用别人的形态定义自己**（划时代产品不以竞品形态自我定义）。vision §2 line 39 给出重定性——终点不是「更好的新闻地球仪」，而是**仪物（instrument-object）**：信息退回环境，产品成为你房间里一件安静的物。R1 = 把这个自我定义写进事实源首句，否则 R5/R7 文案引用「世界表」词根时事实源自相矛盾（REV-018 §1.4）。

### 1.3 建议——首句候选（**★ 三个候选，首句待用户拍板 ★**）

2. **忠于愿景、不自造叙事**：三候选与北极星锚的每一句都能追溯到 product-vision 具体行号（§3/§2/§8），无 vision 之外的新叙事。

## 2. R10 — §9 追加产品宪法（永不引入清单）

### 2.1 现行（`doc/spec.md` §9 line 235–242）

```
## 9. 非目标

- 阶段一零服务器，唯一例外 = 匿名遥测心跳（SPEC-8.7）；无账号系统/社交功能/多端同步。任何产品功能上服务器须过阶段二触发条件…
- 不承诺 iOS 后台常驻刷新与远程推送。
- ACLED 冲突专库需注册，默认不接（冲突类由 GDELT 判类兜底）；接入作为可选增强另行提案。
- 历史事件回放后置（预留缓存窗口 SPEC-6.3，时间滑块 UI 不进早期版本）；全文检索仍非目标…
- AI 摘要/聚类/区域周报后置为付费能力（阶段二前后，SPEC-8.5）。自研 NER 地理解析非目标…
```

**语义注意**：§9 现有各条多为**「后置 / 可提案 revisit」**（阶段二触发、可选增强另行提案）。产品宪法的性质**不同**——是**「永不引入」的负向约束（不设翻案通道）**。故追加时须显式区分，避免被误读为「后置 = 以后可能做」（否则 QA/DEV/后续 arch 可能当作可 revisit 的路线项）。

### 2.2 建议——追加产品宪法条款（在 §9 末尾新增一条 bullet）

> **产品宪法·永不引入（world-watch constitution）**：以下注意力经济器官**写死为永不引入**，**不适用本节其它条目的「阶段二/另行提案」翻案通道**（本条是负向人格约束，非后置路线）——
> ① **无限滚动 / 信息瀑布流**：内容呈现有界，不做无尽下拉喂料；
> ② **算法推荐流**：不按预测参与度/停留时长排序推送内容；世界按用户牵挂组织，不按平台重要性（vision 论纲三）；
> ③ **停留时长 / DAU / 打开率优化**：KPI 倒置——不优化任何延长使用的指标，只优化「合上它之后是更平静还是更焦虑」（vision 论纲二、§8-3；此判据无法埋点，靠人验收）；
> ④ **红点角标 / 未读计数徽章**：不制造「有 N 条未读」的完成焦虑；
> ⑤ **BREAKING 横幅 / 全屏插报**：不以「快来看」的打断攫取注意力（vision 论纲一：表非报）；
> ⑥ **假实时感**：不为「显得活跃/实时」而制造动效或 LIVE 标签；新鲜度须诚实呈现（数据几分钟前就说几分钟前，与 D27/SPEC-3.11a 稳态无动画、vision §4.2 新鲜度诚实一致）。
>
> 依据：product-vision §4.1「有界性入宪」R10、§2 划时代定性（这些器官是注意力经济的结构性器官，买断+零服务器让本品类可合法拒绝它们）。

**逐项 gloss 的性质说明**：每项括注均为**负向约束的边界澄清**（说明「禁什么」），不定义任何新的对外可见正向行为；其中 ④⑤⑥ 均可回落到已有 spec / 决议（D27 常驻零动画、SPEC-3.11a reduced-motion 稳态无动画、vision §4.2），非本卡新立行为。符合 §9 非目标章节体例。

### 2.3 可选——通知宪法负向预置（R6 负向文本，执行承接 M4/FM-19）

REV-018 §5-E 裁定：R6 通知宪法的负向文本**可选**随 R10 §9 批预置、或全随 FM-19（M4）落地——交 orch 定，合拍板 A（真·系统通知归 M4）。orch 已倾向预置。若纳入，建议在 §9 追加**第二条 bullet（仅负向、明确执行承接 M4）**：

> **通知宪法·负向底线（预置，正向机制承接 M4/FM-19/SPEC-8.3）**：Worlens 的通知系统（M4 起，SPEC-8.3）自设计起受两条负向宪法约束——① **默认全静音**（不预设任何默认开启的打断）；② **永不发「回来看看」类召回通知**（不为拉回用户而主动打断，vision §4.1 R6「宁可漏报，不可骚扰」）。**通知的正向发送条件（守望命中 + 高 severity 门槛 + 每日预算上限）属对外可见交互行为，由 M4/FM-19 落 SPEC-8.3 正文定义，本条不预定义**（避免在无通知机制的 M3 阶段行为泄漏）。

**scope 护栏**：本条**只预置负向底线**（永不引入类），**不预置正向门槛机制**（守望命中/severity/预算 = M4 SPEC-8.3 territory，属对外可见行为，本卡不碰）。若 orch 选择不预置，则 R6 全部随 M4/FM-19 落地，本卡仅保留 §2.2 主宪法（无碍）。

### 2.4 理由

1. **宪法 = 负向护城河的显式落钉**：vision §2「被禁锢的品类在买断+零服务器结构里合法了」——永不引入清单是「商业模式即护城河」在 spec 层的负向锚点，与 R1（正向自我定义）成对，同为 D31 叙事/宪法批。
2. **防翻案通道混淆**：显式声明「不适用阶段二/提案翻案」，杜绝被当作可 revisit 的后置项——这正是 §5.2「判据不得向实现看齐」精神的负向版：宪法项不因未实现而软化。
3. **零 src 增量、零新增场景**：纯负向约束文本，无 DEV 实现、无 QA 断言（§3.2 详述）。
4. **通知预置的克制**：只预置永不引入类负向底线，正向门槛留 M4，兑现行为泄漏纪律。
