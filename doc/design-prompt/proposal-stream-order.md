## 0. 问题陈述

现行 SPEC-2.2a 排序句：**「列表按 `ts` 倒序（最新在上，与『事件流』语义一致）」**。该规则默认所有事件都是「过去」事件——`ts` 越大＝越新＝越靠上，对新闻/地震/灾害这类**已发生**事件完全正确、直觉。

但 `ts` 语义并非全体单向过去：**LL2 火箭发射 `ts = net`（T-0 时刻，SPEC-5.5），是未来时刻**。于是同一列表里混入了两类时间语义相反的条目：

| 类型 | `ts` 取值 | 紧迫度随时间的方向 |
| --- | --- | --- |
| 过去事件（news/quake/disaster…） | `ts ≤ now`，越接近 now 越新 | **越久远越不紧迫**（5 分钟前 > 2 小时前 > 1 天前） |
| 未来事件（launch 倒计时） | `ts > now`，越接近 now 越临近发生 | **越临近越紧迫**（T-1h > T-12h > T-9d） |

`ts` 倒序对这两类的作用**恰好相反**：对过去事件「大 `ts` = 紧迫」成立，对未来事件「大 `ts` = 最不紧迫（最远的发射）」却把它顶到首屏第一行。

**实证症状（BUG-024）**：`limit=10` 的 LL2 upcoming 可返回最多 10 条未来发射，`net` 跨度从数分钟到数周。`ts` 倒序下这 10 条按「越远越靠上」排列，「9 天后」的发射稳占列表顶端，而**几分钟前的新地震、T-12h 内即将发射**这些真正紧迫的条目被挤到列表中部——首屏黄金位被最不紧迫的信息占据。

**根因定性**：SPEC-2.2a 的排序规则是在「事件流＝纯过去时间轴、最新在上」的隐含前提下写的，未考虑 `ts` 可为未来时刻（SPEC-5.5）。这是**规格级设计缺陷**（对外可见行为定义错误），走 CLAUDE.md §7 spec 修改路径。

**判据（本提案要满足的目标）**：列表顺序应使**首屏位置与信息紧迫度一致**——离「当前时刻」越近（刚发生的过去 / 临近发生的未来）越靠前；含空态/全未来/全过去边界均有确定行为。呼应 D23 L0「多新」编码意图（首屏应给最「活」的信息，而非最远的未来）。

## 1. 方案选择（不默选，逐一取舍）

### 1.1 建议：时间邻近度排序（`|ts − now|` 升序，双向收敛到 now）

排序主键为**事件时间到当前时刻的绝对距离** `|ts − now|`，**升序**——离 `now` 越近越靠前，无论它在 `now` 之前（刚发生）还是之后（临近发生）。

- **过去事件**：`|ts − now| = now − ts`，越新（越接近 now）距离越小、越靠前 → **等价于现行 `ts` 倒序**。
- **未来事件**：`|ts − now| = ts − now`，越临近发生（如 T-1h）距离越小、越靠前；越遥远（T-9d）距离越大、越靠后 → **倒计时序**。
- **混合**：两侧同时向 now 收敛，最「活」的条目（刚发生 or 即将发生）浮到顶部，最远的未来发射沉底。

**并列消解（保确定性，供 QA 断言）**：`|ts − now|` 相等时，**未来事件（`ts > now`）先于过去事件（`ts ≤ now`）**（临近发生的事件仍可被观看/等待，动作性略高于已定格的过去）；仍并列者按 `id` 升序稳定排序。连续时钟下精确等距为测度零、视觉上几乎不出现，但须定义以保 DEV/QA 不发散、测试可确定断言。

**为何满足判据**：
- 紧迫度 = 时间邻近度（离 now 越近越「活」），这是**单一全序**、无分区拼接缝隙，任何两条事件都有确定先后。
- **边界优雅退化**（三种边界都被同一规则覆盖，无需特例分支）：
  - **全过去** → `ts` 倒序（最新在上）＝**现行行为不变**，对绝大多数场景（news/quake/disaster 全过去）零回归、保「事件流」直觉。
  - **全未来** → 最近发生者在上＝纯倒计时序，正是发射清单的直觉序。
  - **空态** → 列表为空，落 SPEC-2.2a 现有空状态文案子句（本提案不改该子句，见 §3）。
- **动态语义自洽**：`now` 随时间推进——过去事件整体等速远离、相对序不变（最新恒在顶）；未来发射随倒计时**向顶部攀升**，越过 T-0（发射发生）后转为过去事件、随年龄**向下漂移**。这与 D9 呼吸式过渡/时间流动的直觉一致，且排序基准时刻与③相对时间同源（都基于 `now`），列表位置与每行显示的相对时间不会互相矛盾。

**代价（诚实列出）**：混合区在 now 两侧近似对称时会出现「前/后」交替的相邻行（如「10 分钟后 / 5 分钟前 / 30 分钟后」）。但——① 每行自带相对时间标签、自解释，交替不产生歧义；② 交替仅在两条事件都紧邻 now（都很「活」）时出现，此时它们**本就同等紧迫**，相邻是合理的；③ 「只想看发射」的用户走 SPEC-2.4 六分类过滤到 launch 后，本规则退化为纯倒计时序。故该代价可接受、且被既有分类过滤架构兜底。

### 1.2 替代方案及否决理由

| 方案 | 机制 | 否决理由 |
| --- | --- | --- |
| **A. 保持 `ts` 倒序（现状）** | 全体按 `ts` 大到小 | 即 BUG-024 本身：未来发射大 `ts` 顶首屏，最不紧迫者占黄金位，与判据直接相悖。**否决**。 |
| **B. 硬分区：全未来区置顶 + 全过去区其后（或反之）** | 先按 `ts>now / ts≤now` 分两段，段内各自排序 | 分区边界不在 now，**不满足紧迫度对齐**：未来在上时，「9 天后」的发射仍压在「2 分钟前」的 M6 地震之上；过去在上时，「T-1h」的临近发射被压在「1 天前」的旧事件之下。把「未来 vs 过去」当排序主轴，而真正的主轴是「离 now 远近」。**否决**。 |
| **C. 分区 + 段内向 now 收敛（未来段升序 ts、过去段降序 ts，两段在 now 处对接）** | 未来段最临近在段尾、过去段最新在段首，中缝＝now | 若把中缝拼成一条列表，其顺序其实**就是方案 A 建议的 `|ts−now|` 升序的一个特例排布**（只是用两段实现），但强行分区会在「未来最远」与「过去最新」之间产生一条不自然的硬缝，且需额外定义两段谁在上。方案 1.1 用单一 `|ts−now|` 键即得同样的「向 now 收敛」效果、无缝、无额外分段规则。**以其为更简形式否决独立分区实现。** |
| **D. 按 severity 排序（或 severity 为主键、时间为次键）** | 高 severity 优先置顶 | ①**越界**：行内 severity 表达属另一张卡（BUG-025），把排序绑到 severity 会耦合两卡、并与 BUG-023 入球阈值纠缠，违反本卡「只解决排序」的边界；②severity 只有三档，档内仍需时间排序才能定序，并未解决本卡的时间语义问题；③紧迫度 ≠ severity（一条 severity 1 的临近发射 vs 一条 severity 1 的 9 天后发射，仍需时间邻近度区分）。**本卡否决 severity 排序**；severity 若未来进入排序仅作为可能的更高优先级键，属 BUG-025 卡的联动议题，见 §7。 |
| **E. 未来事件单独置顶「倒计时区」+ 过去事件下方「信息流区」并各设小标题** | 视觉分区 + 标题 | 引入了「分区标题」这一**新的对外可见 UI 结构**，超出「排序语义」范围、且与 SPEC-2.2a 现有「单一列表」结构冲突，属更大的面板信息架构改动。本卡只重定义顺序、不新增 UI 元素。**否决（超范围）**。 |

**结论**：方案 1.1（`|ts − now|` 升序）以**单一全序**同时满足「首屏＝最紧迫」「三边界优雅退化」「全过去零回归」「与相对时间同源自洽」，且不越界到行内容/severity。这是独立论证结果——BUG-024 摘要提到的「绝对差双向收敛」被采纳，「未来过去分区」（方案 B/E）经论证否决。

## 2. 边界情形逐一覆盖（任务点名三边界）

| 边界 | 输入 | 本规则行为 | 与判据一致性 |
| --- | --- | --- | --- |
| **空态** | 无事件 | 列表为空 → 落 SPEC-2.2a 现有空状态文案「暂无事件」（本提案不改该子句） | 无条目，无排序问题；空态文案保持 |
| **全过去** | 所有 `ts ≤ now`（news/quake/disaster 典型态） | `|ts−now|` 升序 ≡ `ts` 倒序，最新在上 | **等于现行行为，零回归**，保「事件流」直觉 |
| **全未来** | 所有 `ts > now`（例如只加载了 LL2） | `|ts−now|` 升序 ≡ 最近发生者在上，纯倒计时序 | 首行＝最临近发射，最紧迫在上 |
| **混合** | 过去 + 未来并存（BUG-024 场景） | 两侧向 now 收敛，最「活」者在顶，最远未来沉底 | 首屏＝最紧迫（刚发生 or 即将发生） |
| **恰在 now** | `ts = now`（`net` 刚好等于当前时刻） | `|ts−now|=0`，排最前；与另一 `ts=now` 并列时按 §1.1 并列规则（`ts>now` 优先，`ts=now` 归过去侧，再 `id`） | 极端瞬时点，有确定行为、不 NaN/不抖动 |

## 3. SPEC-2.2a 修订文本（现行 / 建议 / 理由）

### 现行（`doc/spec.md` SPEC-2.2a，整条，便于 pin 替换）

> - **SPEC-2.2a** 事件流面板列表行：SPEC-2.2 事件流面板的每个列表行显示三项——① 分类色圆点（取 SPEC-3.7 对应 category 色）；② 事件标题（SPEC-6.1 `title`）；③ 相对时间（由 SPEC-6.1 `ts` 相对当前时刻派生，如「3 分钟前」，格式细节属实现自由度）。列表按 `ts` 倒序（最新在上，与「事件流」语义一致；`ts` 仅供展示/排序，SPEC-6.3①）。无事件时显示空状态文案「暂无事件」。severity、地点、摘要、信源等完整字段属详情卡（SPEC-2.3，M3），不进列表行。

### 建议（**仅替换排序那一句**；①②③、空状态、详情卡三句逐字保留，此处引全条便于 pin 替换）

> - **SPEC-2.2a** 事件流面板列表行：SPEC-2.2 事件流面板的每个列表行显示三项——① 分类色圆点（取 SPEC-3.7 对应 category 色）；② 事件标题（SPEC-6.1 `title`）；③ 相对时间（由 SPEC-6.1 `ts` 相对当前时刻派生，如「3 分钟前」，格式细节属实现自由度）。**列表按「距当前时刻 `now` 的时间邻近度」升序排序（`now` 与③相对时间同源、随之刷新）：主排序键为 `|ts − now|`，值越小（离当前时刻越近——无论刚发生的过去事件，还是临近发生的未来事件如发射倒计时）越靠前，值越大（越久远的过去或越遥远的未来）越靠后；`|ts − now|` 相等时未来事件（`ts > now`）先于过去事件（`ts ≤ now`），仍并列者按 `id` 升序稳定排序以保确定性。全部为过去事件时该规则等价于 `ts` 倒序（最新在上，保留「事件流」直觉）；全部为未来事件时等价于最近发生者在上（倒计时序）。`ts` 仅供展示/排序、不参与过期判定（SPEC-6.3①）。** 无事件时显示空状态文案「暂无事件」。severity、地点、摘要、信源等完整字段属详情卡（SPEC-2.3，M3），不进列表行。

### 理由

- 修掉 BUG-024 根因：`ts` 倒序对未来事件（`ts=net`，SPEC-5.5）方向相反，把最远发射顶到首屏。改为「离 now 越近越靠前」，使首屏位置与紧迫度对齐（呼应 D23 L0「多新」）。
- **不动 `ts` 语义、不改 SPEC-5.5/6.1/6.3**：`ts` 仍是事件时间、仅供展示/排序（本条末句原样保留 SPEC-6.3① 交叉引用）。LL2 `ts=net`（REV-008 pin）、severity 方向（REV-009，BUG-019 pin）均正交、无需改。本提案只改「列表怎么用 `ts` 排序」，不改「`ts` 是什么」。
- **三边界收敛进同一规则**：空态落原空状态子句、全过去退化为现行 `ts` 倒序（零回归）、全未来退化为倒计时序，无需特例分支。
- **①②③、空状态文案、详情卡归属三句逐字不动**——它们属列表行**内容**定义（BUG-025 卡范围），本提案不触碰（§6 合并须知）。

> **行为泄漏禁区自查（arch.md 硬规则）**：排序是**对外可见行为**，本提案把完整排序规则（含并列消解与三边界退化）全部写入 §3 的 SPEC-2.2a 修订正文（走 spec 修改路径），**不**把任何排序语义只留在 design-prompt。DEV 侧仅承接「按该 spec 键计算顺序」的实现自由度（如比较器写法、重排刷新频率），不承接任何未 pin 的可见顺序决策。

## 4. 极简边界（本提案不做）

- **不改列表行的可见内容**（①②③、空状态文案、severity/摘要归属）——BUG-025 卡范围。
- **不引入 severity 排序键**——见 §1.2 方案 D 否决理由；severity 是否成为更高优先级键属 BUG-025 联动议题（§7）。
- **不新增分区/小标题等 UI 结构**（方案 E 否决）；列表仍是单一列表。
- **不规定重排刷新频率/比较器实现**——属 DEV 实现自由度，只锁「排序基准时刻与③相对时间同源」这一对外可见约束。
- **不改 SPEC-5.5（`ts=net`）/SPEC-6.1（`ts` 定义）/SPEC-6.3①（`ts` 不参与过期）**——三者与本提案正交。
- **不处理 BUG-023 入球阈值**（M<2 微震淹没）——那是「哪些事件进列表/进球」的过滤问题，非「进了之后怎么排」的排序问题，属独立缺陷。

### 5.2 testplan（qa）

**M2-13 复测口径**（现 ✅ v0.2.5 → 其中排序断言改写、须重测）：

- **不变的判据半**：三要素齐备（分类色圆点+标题+相对时间）、不额外渲染 severity/摘要、空状态文案「暂无事件」、面板 300px/可折叠——**均保留**（这些属行内容与布局，未受本提案影响）。
- **改写的判据半**：原「列表按 `ts` 倒序（最新在上）」→ 改为「列表按 `|ts − now|` 升序（离 now 越近越靠前，含 §1.1 并列规则）」；断言引用修订后的 SPEC-2.2a。
- **新增覆盖点（锁 BUG-024 根因 + 三边界，qa 从 spec 推导，可并入 M2-13 或另立场景 qa 定夺）**：
  - **混合排序**：构造「一条未来发射（`ts=now+若干天`）+ 一条刚发生过去事件（`ts=now−几分钟`）」，断言过去事件在未来发射之上（现行 `ts` 倒序下相反）——直锁 BUG-024。
  - **全未来退化**：仅未来发射多条（`ts=now+1h / +12h / +9d`），断言最临近者（+1h）在顶、最远者（+9d）沉底（倒计时序）。

## 7. 遗留风险 / 待仲裁项

1. **severity 是否应为更高优先级排序键**：本卡按边界否决 severity 排序（§1.2 D），只做时间邻近度。但 BUG-025 卡将引入行内 severity 编码后，产品上可能希望「高 severity 无条件靠前，时间邻近度为次键」。这是**跨两卡的联动议题**，本卡不拍板、不预埋——若未来需要，应由 BUG-025 卡或独立提案在两卡都落定后统一论证（避免本卡越界耦合）。已列此风险供 rev 知悉。
2. **重排视觉churn**：`now` 推进使未来发射持续向顶攀升、跨 T-0 后转过去下漂，理论上会引起列表行位置随时间移动。属预期行为（与相对时间刷新同源），但若实测在高频刷新下出现视觉抖动/闪跳，属实现层（刷新节流）问题，非 spec 问题，届时登记 DEV 侧优化，不改本排序语义。
3. **依赖裁决**：本提案须经 rev 仲裁通过后，由 orch 应用 SPEC-2.2a 正文（句级替换）+ §0 修改记录 + `make pin-spec` + 同步 M2-13/M2-globe.md，dev 方可改面板比较器。在此之前 dev 不得抢改排序（现「`ts` 倒序」保持，M2-13 现 ✅ 暂不动）。
