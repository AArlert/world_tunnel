## 0. 问题陈述（D22 门禁与待 spec 化的五点）

D22 定 GDELT/RSS 等**报道类**信源接入（M3）前，必须先 spec 化「最小信任层」。当前 spec 缺口：

1. **信源分级缺失**：SPEC-6.1 `GeoEvent` 只有 `source`（源短码枚举），**无任何字段或契约表达「权威事件源 / 新闻报道（待验证）」两级**。M2 现有 T1（USGS/EONET/GDACS/LL2）都是权威源，M3 起 GDELT（及 M6 RSS）是报道类——报道类接入前必须先有分级契约，否则一条未经证实的新闻标记与一条 USGS 地震在详情卡里等价呈现，违背 D1「可信」核心判据。
2. **发布/更新时间**：D23 L1 明列详情卡须给「发布/更新时间」；SPEC-2.3 现有「时间（相对+绝对）」但未明确其字段来源与对报道类信源的语义（`ts` = 事件时间，见 proposal-expiry-semantics.md）。
3. **去重呈现**：D22 举例「多源报道 N 条」。SPEC-5.3/5.5 已有「同 `eventid` 分组去重」，SPEC-2.3 已有「信源链接列表（≥1 条）」——须厘清「多源报道 N 条」是这些既有机制的延伸，还是新的跨源聚类。
4. **轻量纠错反馈入口**：D22 定 mailto/仓库 issue 级、零服务器（呼应 D16/D18、SPEC-8.7/§9）；SPEC 无对应元素，须落章节。
5. **T1 不受影响的验证**：D22 断言「T1 权威源阶段（M2）不受影响」；本提案设计须验证该句成立（不要求 M2 provider 补任何新字段），否则显式指出。

本提案对每点给出设计判断（含选项与取舍）、SPEC 修订文本、影响面，并显式验证第 5 点。

## 1. 设计决策（逐问，含选项与取舍，不默选）

### 1.1 【Q1】信源分级 = 与 SPEC-5.8 T1–T4 解析分层**正交独立轴**，且**不新增 GeoEvent 字段**（建议 Design A）

**判断一：信任分级与 SPEC-5.8 的 T1–T4 是两条正交轴，必须独立建模，不得互相推导。**

- SPEC-5.8 T1–T4 回答「**如何得到坐标**」（自带坐标 → gazetteer 查表 → GDELT 地理编码 → 人肉钉图）——是**定位解析策略**。
- 信任分级回答「**事件可信度 / 来源性质**」（一手系统级记录 vs 二手报道）——是**来源可信性分类**。
- 二者**相关但不同构，强行合并会产生范畴错误**：
  - 反例 A：**OpenSky 航班**（SPEC-5.6）自带 ADS-B 航迹坐标（定位上属「自带坐标」），是权威遥测源，但**不在 SPEC-5.8 的 T1 枚举（T1=USGS/EONET/GDACS/LL2）内**——证明「自带坐标」≠「T1」≠「权威」三者并不等号。
  - 反例 B：未来某官方机构 feed 可能只给结构化地名、需 T2 gazetteer 解析（定位属 T2），但仍是权威源；某自定义 RSS 可能内嵌 `geo:` 坐标（定位近似 T1），但仍是报道类（待验证）。
  - 故：把「信任级」折进 SPEC-5.8 会锁死「权威源必自带坐标 / 报道源必需地理编码」这条并不成立的隐含约束。**两轴独立**。

**判断二：分级是「信源级常量」，用 `source → 分级` 对照表派生，不新增 GeoEvent 字段（Design A）。**

| 选项 | 机制 | 取舍 |
| --- | --- | --- |
| **A（建议）**：新增 SPEC-5.10 信源→（显示名+信任等级）对照表，UI 由 `source`（SPEC-6.1）派生，**不改 GeoEvent** | 分级是信源级常量（非逐事件从 payload 归一化产出），与 category→颜色（SPEC-3.7 由 `category` 经表派生、不入模型）同构 | **① 最小**：分级是 `source` 的纯函数，存字段即冗余。**② 与既有先例一致**（SPEC-3.7 颜色不入模型）。**③ 兑现 D22「T1 不受影响」**（见 §1.5）：M2 provider/store/归一化零改动、M2-01 结构断言不动。取「模型干净 + 契约进表」。 |
| B（否决）：给 `GeoEvent` 加 `tier` 字段，各 provider 归一化时打常量 | 每个 normalizer 盖一个自己的常量 tier | 冗余于 `source`；且对 M2 provider 逐个改代码、M2-01 须重测——**直接违反 D22「T1 不受影响」**（代码被动、行为虽未变仍是改动）。这与 category 必须是逐事件字段的情形不同：category 有 GDACS→disaster/humanitarian、GDELT→news/conflict 的**逐事件多值**，而 tier 在 M3 范围内**信源级恒定**（GDELT 全待验证、权威源全权威），无逐事件变化 → 无理由入模型。**否决**。 |

**结论**：Design A——新增 **SPEC-5.10 信源信任分级**（数据契约，§5 章）+ 由 `source` 派生，`GeoEvent` 不改。分级消费方仅 L1 详情卡（SPEC-2.3）。

两级命名沿用 D22 原文（**用户可见字符串，须 pin**）：**「权威事件源」/「新闻报道（待验证）」**。

M3 范围内 `source → 分级`：

| source | 显示名（信源名） | 信任等级 |
| --- | --- | --- |
| `usgs` | USGS | 权威事件源 |
| `eonet` | NASA EONET | 权威事件源 |
| `gdacs` | GDACS | 权威事件源 |
| `ll2` | Launch Library 2 | 权威事件源 |
| `opensky` | OpenSky | 权威事件源 |
| `gdelt` | GDELT | 新闻报道（待验证） |

- **CoinGecko 行情 ticker（SPEC-5.7）非 GeoEvent（仅顶栏），不入本表**。
- 未来**自定义 RSS 信源（SPEC-5.9，M6）一律归「新闻报道（待验证）」**，显示名取 feed 标题（前向兼容，M6 落地）。
- 新增/变更任一信源的分级属 §7 spec 变更。

### 1.2 【Q2】发布/更新时间 = 复用 `ts`，**不新增独立时间字段**（建议）

| 选项 | 机制 | 取舍 |
| --- | --- | --- |
| **A（建议）**：详情卡「发布/更新时间」= 既有 `ts`（SPEC-2.3 已列「时间（相对+绝对）」）；报道类信源的 `ts` 即其报道发布/更新时刻 | 对权威源 `ts` = 事件时间（origin/T-0）；对报道类 `ts` = 报道发布时刻（GDELT `seendate` 类，SPEC-6.1「无则用抓取时间」兜底）；SPEC-6.3 首句「同 id 更新覆盖 `ts`」使 `ts` 天然也是内容「最后更新时刻」 | **每事件只有一个时间**：权威源无独立「发布时刻」概念，报道类 `ts` 已 ≈ 发布时刻；再加字段对多数源为空/冗余。D23 L1「30 秒预算」也主张**一个时间而非两个**。满足 D23「发布/更新时间」无需新字段。 |
| B（否决）：新增 `publishedAt` 区分于 `ts` | 同时存事件时间与发布时间 | 权威源 `publishedAt` 恒等/空于 `ts`；报道类 `ts` 已是发布时刻——加一个多数场景等于 `ts` 的字段污染模型。仅当须**同屏并陈「事件发生于 X / 报道于 Y」**才成立，而 D23 L1 不要（一个时间）。**否决**。 |

- **`lastSeen` 不参与、不展示**：`lastSeen`（proposal-expiry-semantics.md 定案）是存储层过期记帐、墙钟-最后-upsert 时刻，对用户无意义，**详情卡不显示**。D23「更新时间」由 `ts` 在 SPEC-6.3 更新语义下的推进承载，与 `lastSeen` 无关。
- 因此 SPEC-2.3 只需**明确其时间元素 = `ts` 且注明报道类语义**，不动 SPEC-6.1。

### 1.3 【Q3】去重呈现 = 既有 `urls[]` 链接列表（单事件多信源计数），**跨源语义融合不做**（属 D15 付费阶段二）

**三种机制须分清：**

1. **同 `id` 身份去重**（SPEC-5.3 GDACS 按 `eventid` 分组 / SPEC-6.3 首句「同 id 更新」）：同一逻辑事件的多要素/多轮返回 → 收敛为**一个 GeoEvent**。这是**身份去重**（去重键 = `id`，SPEC-6.1）。
2. **单事件多信源链接呈现**：一个 GeoEvent 的 `urls[]`（SPEC-2.3「信源链接列表（≥1 条）」）承载该事件的多个信源链接；`urls.length > 1` 时即「多源报道 N 条」的**最省实现**——**这就是 D22「去重呈现」的既有雏形，已在 SPEC-2.3 隐含**（任务假设成立）。
3. **跨源事件聚类/融合**：不同 `source`（→ 不同 `id`，SPEC-6.1 `{source}:{原始id}`）对**同一真实事件**的多条独立报道 → 聚为一枚标记。既有 `id` 去重**做不到**（跨源 id 不同）。这是**真正新的机制**。

**判断：本「最小信任层」只做机制 2，不做机制 3。**

- 机制 3（跨不同真实报道的语义融合/聚类/清洗）正是 **D15 明确后置为付费阶段二的「融合/去重/清洗」AI 消化能力**（「前提是数据层先做扎实」「早期核心 = 解析地点与类型、球面显示、点击查看信源内容」），并受 §9「AI 摘要/聚类后置」约束。M3 不建聚类引擎，与「最小信任层」的「最小」自洽。
- 故 D22「多源报道 N 条」在 M3 落为：**单个 GeoEvent 的 `urls[]` 条数**（`urls.length`），于 L1 详情卡呈现；不新增跨源聚类。
- **层级归属**：「N 条报道」是 L1（详情卡）元素，**不上 L0**——D23 L0 只编码位置/分类/新鲜度三变量、明禁编码信源（NewsGlobe 反例）。故 SPEC-2.2a 列表行与球面标记**不显示信源计数**。

### 1.4 【Q4】纠错反馈入口 = 落 SPEC-2.3 详情卡（L1），mailto/issue 级、零服务器

- 纠错对象是**具体事件**（其定位/分类/真伪存疑）→ 天然按事件、归 **L1 详情卡（SPEC-2.3）**，而非全局设置项。
- 契约（须 pin）：详情卡提供一处轻量反馈入口，点击打开 **mailto: 或仓库 issue 链接**，预填事件 `id`/标题/信源以便定位；**零服务器**、不发起任何服务端提交（呼应 SPEC-8.7「阶段一唯一服务端组件 = 遥测心跳」、§9 零服务器、D18 隐私）。
- **实现自由度**（不进 spec、不展开）：mailto 或 issue 二选一、按钮文案、链接模板等；spec 只 pin「零服务器 + 预填事件上下文 + 打开外部 mail/issue」这一对外契约（本卡不涉及"用什么库发邮件"等实现，纠错入口只到「mailto/issue 链接」级）。

### 1.5 【Q5】验证 D22「T1 权威源阶段（M2）不受影响」——在 Design A 下**成立**

逐项核验本提案对 M2 已实现子句（SPEC-5.1/5.2/5.3/5.5、SPEC-6.1、SPEC-6.3、SPEC-2.2a）的触碰：

| 提案要素 | 落点 | 对 M2 影响 |
| --- | --- | --- |
| 信源分级（SPEC-5.10 新增） | 数据契约表 + 由 `source` 派生 | **零**：不改 SPEC-6.1，M2 provider 不补字段、归一化不改，M2-01 结构断言不动；表消费方（详情卡）是 M3。 |
| 发布/更新时间（复用 `ts`） | SPEC-2.3 时间语义澄清 | **零**：不改 SPEC-6.1/6.3，`ts` 语义不动（proposal-expiry-semantics.md 已定）。 |
| 去重呈现（`urls[]` 计数） | SPEC-2.3（M3 详情卡） | **零**：`urls[]` 是 SPEC-6.1 既有字段；计数呈现在 M3 详情卡，M2 无详情卡。 |
| 纠错入口 | SPEC-2.3（M3 详情卡） | **零**：M2 无详情卡。 |
| L0/列表不编码分级 | SPEC-2.2a **不改**（确认） | **零**：D23 L0 已禁信源编码，M2 列表行现状即不显示信源，无需改。 |

**结论：D22「T1 不受影响」在 Design A 下成立**——全部落点集中在 M3 详情卡（SPEC-2.3，FM-14）与一张新数据契约表（SPEC-5.10），不触碰任何 M2 代码/字段/已 ✅ 场景判据。**这正是 Design A 优于 Design B 的决定性理由**：若取 Design B（GeoEvent 加 `tier` 字段），M2 provider 须逐个盖 tier、M2-01 须重测——D22 该句将被**边际违反**（行为虽未变但 M2 代码被动），故取 A。

## 2. SPEC 修订文本（现行 / 建议 / 理由）

### 2.1 新增 SPEC-5.10 信源信任分级（trust tier）

**现行**：无。

**建议（新增于 §5，紧随 SPEC-5.9 之后）**：

> - **SPEC-5.10 信源信任分级（trust tier，M3）**：每个 GeoEvent 信源归入两级之一，用于 L1 详情卡（SPEC-2.3）明示事件来源性质。**与 SPEC-5.8 的 T1–T4 解析分层为正交独立轴**——SPEC-5.8 定「如何得到坐标」，本条定「来源可信性质」，二者不得互相推导（如 OpenSky 自带航迹坐标却不在 T1 枚举、仍属权威）。
>   - **权威事件源**：信源本身即事件的系统级记录（传感网/官方机构/发射登记/航迹遥测/行情聚合）。
>   - **新闻报道（待验证）**：对事件的二手报道，事件真伪与表述依赖第三方报道而非一手记录。
>   - **信源 → 显示名 + 信任等级**（用户可见字符串以本表为准；新增/变更任一信源分级属 §7 spec 变更、须以来源性质为据）：
>
>   | source | 显示名 | 信任等级 |
>   | --- | --- | --- |
>   | `usgs` | USGS | 权威事件源 |
>   | `eonet` | NASA EONET | 权威事件源 |
>   | `gdacs` | GDACS | 权威事件源 |
>   | `ll2` | Launch Library 2 | 权威事件源 |
>   | `opensky` | OpenSky | 权威事件源 |
>   | `gdelt` | GDELT | 新闻报道（待验证） |
>
>   - 分级为**信源级常量**（非逐事件从 payload 归一化产出），由 `source`（SPEC-6.1）经本表派生，**不新增 GeoEvent 字段**（沿用 SPEC-3.7「颜色由 `category` 经表派生、不入模型」先例）。
>   - CoinGecko 行情 ticker（SPEC-5.7）非 GeoEvent（仅顶栏），不入本表；未来自定义 RSS 信源（SPEC-5.9，M6）一律归「新闻报道（待验证）」，显示名取 feed 标题。
>   - 分级**仅用于 L1 详情卡（SPEC-2.3）**；**L0 球面标记与事件流列表行（SPEC-2.2a）不编码信源/分级**（D23 L0 只编码位置/分类/新鲜度三变量）。

**理由**：兑现 D22 两级分级门禁；与 SPEC-5.8 显式解耦，避免范畴错误；采「由 `source` 派生、不入模型」的最小形态，与 SPEC-3.7 先例一致，并使 M2 零改动（§1.5）。

### 2.2 改写 SPEC-2.3 事件详情卡

**现行**：

> - **SPEC-2.3** 事件详情卡：点击球面标记或列表条目弹出浮层——标题、分类徽章、severity、时间（相对+绝对）、地点、摘要、信源链接列表（≥1 条）。

**建议**：

> - **SPEC-2.3** 事件详情卡（M3，L1，D23）：点击球面标记或列表条目弹出浮层——标题、分类徽章、severity、时间（相对+绝对）、地点、摘要、**信源名与信任等级（SPEC-5.10）**、信源链接列表（≥1 条）、**轻量纠错反馈入口**。
>   - **时间语义**：所显示时间取 GeoEvent `ts`（SPEC-6.1，事件时间/无则抓取时间），相对+绝对并陈；对**新闻报道（待验证）**类信源，`ts` 即该报道的发布/更新时刻（各 provider 的 `ts` 映射见 SPEC-5.x），据此满足 D23 L1「发布/更新时间」，**不新增独立时间字段**；存储层内部的 `lastSeen`（SPEC-6.3①）为过期记帐、**不展示**。
>   - **去重呈现**：信源链接列表即单个 GeoEvent 的多信源链接（≥1 条）；`urls` 多于 1 条时可呈现其条数（如「N 条报道/信源」）。此为**单 GeoEvent 内的链接计数**，非跨源事件聚类——跨不同真实报道的语义融合/清洗属付费阶段二 AI 能力（D15、§9、SPEC-8.5），本条不做。
>   - **纠错反馈入口**：提供一处轻量反馈入口，点击打开 **mailto: 或仓库 issue 链接**（预填事件 `id`/标题/信源以便定位），**零服务器**、不发起任何服务端提交（呼应 SPEC-8.7、§9 阶段一零服务器、D18 隐私）；具体渠道（mailto 或 issue）与文案属实现自由度，「零服务器 + 预填事件上下文 + 打开外部 mail/issue」为对外契约。

**理由**：把 D23 L1「信源名与等级 + 发布/更新时间 + 原文链接」完整落到详情卡；「去重呈现」明确锚在既有 `urls[]`、划清与 D15 跨源融合的界；纠错入口按事件归 L1、零服务器守 D18；元素新增全部 M3、不触 M2（§1.5）。

### 2.3 SPEC-6.1 GeoEvent 模型

**不改**（决定见 §1.1 判断二、§1.2）。修订文本 = 无。`source` 枚举已含 `gdelt`；分级/信源名/发布时间均不新增字段。

## 6. 遗留风险 / 待仲裁项

1. **建模边界（Design A vs B）**：§1.1 已建议 A（不加 GeoEvent 字段、由 `source` 派生）并给 T1-不受影响的决定性理由；若 rev 判「加 SPEC-6.1 `tier` 字段」更清晰，则须相应改写 SPEC-6.1（加 `tier: 'authoritative' | 'reported'`）并承认 M2 provider 逐个补字段、M2-01 重测的代价（D22「T1 不受影响」边际违反）。此为待 rev 拍板的模型-契约边界取舍，已列取舍。
2. **逐事件分级前向风险**：M3 分级为信源级常量；若未来单一信源混合可信度事件（如某 feed 同时含官方公告与转载新闻），SPEC-5.10 的「信源级」假设失效，须届时另提案引入逐事件分级（可能反过来支持 Design B）。当前范围内不成立，登记为前向风险。
3. **显示名取值**：SPEC-5.10 表内显示名（USGS/NASA EONET/…）为 arch 拟定的用户可见字符串，若 aes/rev 有更佳呈现（如中文全称）可在仲裁时调整；本表以短英文名为默认，属可仲裁项、非阻断项。
4. **「N 条报道」文案与阈值**：SPEC-2.3 定「`urls>1` 可呈现条数」，具体文案（「N 条报道」/「N 个信源」）留实现自由度；若产品要固定文案则须在仲裁时 pin。
5. **依赖裁决**：本提案须经 rev 仲裁通过后由 orch 应用（SPEC-5.10 新增 + SPEC-2.3 改写 + §0 修改记录 + `make pin-spec` + 同步 testplan/FM/DP），dev 方可据此实现 M3 详情卡；在此之前不得抢改。
