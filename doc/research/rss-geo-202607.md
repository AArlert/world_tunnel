## Q1 主流与自托管 RSS 阅读器：是否有「按地图/地理组织 feed」的功能/视图/插件

### ✅ 事实

逐个核查结果——**均未找到任何按地图/地理位置组织 feed 内容的功能、视图或官方插件**；全部产品的组织维度是「订阅源 / 文件夹 / 标签 / 主题」，即传统「按订阅源」模型：

| --- | --- | --- | --- |

| NetNewsWire | 本地（Mac/iOS） | **无**（未找到） | 本地 feed + 文件夹，iCloud 同步；无地图视图。[github.com/Ranchero-Software/NetNewsWire](https://github.com/Ranchero-Software/NetNewsWire/)、[App Store](https://apps.apple.com/us/app/netnewswire-rss-reader/id1480640210) |

### 🔶 推测/判断

- 传统 RSS 阅读器**在结构层面就没有地理维度**：其数据模型是「源 → 条目」的树，条目不携带、也不解析地理坐标；地图组织需要为每条目做「自由文本 → 坐标」的解析，这是阅读器们从未纳入的一层。经过 15 年以上品类演进（Feedly/Inoreader 皆超十年）仍无一家做，倾向说明这**不是被遗漏的小功能，而是与其「读源」定位不合的能力**——正是 D14 修订所指的空位方向。
- 边界：Web 检索非店内/插件市场穷举，个别小众第三方脚本可能漏检；但**七款主流/自托管代表产品**一致为负，信号足够强。

## Q2 GeoRSS / geotagged feed 生态现状

### ✅ 事实

- **GeoRSS 标准仍存，但消费端偏「基础设施」而非「产品」**：GeoRSS 是把位置编码进 Web feed 的规范；其 W3C 序列化「**较旧且部分被弃用（partly deprecated），但仍是使用最广的一种**」，另有 GeoRSS-Simple 与 GeoRSS GML 两种编码；规范目标消费方是「**地图生成器一类地理软件**」，而非面向读者的阅读产品。[Wikipedia: GeoRSS](https://en.wikipedia.org/wiki/GeoRSS)
- **georss.org 已近乎存档式**：站点主要是规范文档与历史示例（引用到 USGS 2004 年代地震告警等旧例），无「当前在营实现」的活指南。[georss.org](https://georss.org/)（抓取 2026-07-21）
- **历史 mashup 工具仍挂网但明显陈旧**：GeoNames「RSS to GeoRSS Converter」读取 RSS 条目文本、在 GeoNames 库中查地名并输出 GeoRSS/KML，示例为「把 Reuters World News 显示在地图上」；页面称**首个已知的 GeoRSS 阅读器是 ACME GeoRSS Map Viewer**。这些是老工具，页面呈存档特征，**当前是否仍正常运行未取得确证**。[geonames.org/rss-to-georss-converter](https://www.geonames.org/rss-to-georss-converter.html)、[acme.com/GeoRSS](https://www.acme.com/GeoRSS/)
- **GeoRSS 作为开发者/GIS 管线仍被支持**：Google Maps JavaScript API 仍文档化 GeoRSS 图层示例；Esri/TIBCO 侧亦有「向地图添加 RSS」文档。即 GeoRSS 活在**开发者集成层**（把 feed 灌进 GIS/地图 SDK），而非独立消费产品。[Google Maps GeoRSS Layer](https://developers.google.com/maps/documentation/javascript/examples/layer-georss)、[TIBCO/Esri RSS feeds](https://docs.tibco.com/appstudio/913/doc/html/topic/com.ibi.help.apps/source/8205_esri_rssfeeds.htm)
- **面向消费者的在营 GeoRSS 阅读/可视化产品——未找到有影响力者**：检索未见任何当前主流、面向普通读者的「GeoRSS 阅读器」产品；该层基本已历史化。

### 🔶 推测/判断

- GeoRSS 的现状可概括为「**标准与管线尚存，消费产品已凋零**」：它作为「让开发者把带坐标的 feed 灌进地图」的格式还活着，但作为「普通人用来按地图读 feed」的形态在过去十余年基本消散——这与 Q1 的负结果同源（阅读器不做地理，地图产品又不做通用阅读）。
- 关键含义：GeoRSS 生态之所以凋零，很大程度是因为**多数 feed 并不自带坐标**（geotagged feed 本身稀少），真正的难点从来在「无坐标文本 → 坐标」的解析，而非编码格式——这正是 D13 分层（T2 gazetteer / T3 GDELT）要解决的问题，也印证 D14 读法的价值不在「消费 GeoRSS」而在「给普通 feed 补地理」。

## Q4 可守性判断：门槛与「谁最可能先补齐」

### ✅ 事实（支撑判断的已核证据）

- 该能力**稀缺但非空白**：完整实例 = NewsGlobe（独立、无成熟度信号、非中文）；邻近实例 = World Monitor（条目级 geo，但仅 curated feed、开源）。传统阅读器（Q1 七款）**零**地理组织。来源见 Q1/Q3。
- 传统阅读器的「按订阅源」模型历十余年未纳地理维度（Feedly/Inoreader 皆 10+ 年），是**结构性缺位**而非临时空档。来源见 Q1。
- 自由文本 → 坐标是公认难点：geotagged feed 本身稀少（Q2），主流路径是「读条目文本、查 gazetteer/NER 定位」（GeoNames converter、NewsGlobe 均如此）。来源见 Q2/Q3。

### 🔶 推测/判断

**补齐该能力的主要门槛（按对 Worlens 的相关度排序）：**
1. **非英文（尤其中文）地理解析**——观察到的实例（NewsGlobe/GeoNames converter/GDELT 系）几乎都以英文/欧语实体抽取为主，**未见对中文 feed 做条目级地理落点者**。蓝点网这类中文源的「欧盟/美国新闻混发」正落在此空档：中文地名 NER + gazetteer + 别名消歧是一层独立工程门槛，也是 Worlens 最可能形成差异的地方。
2. **自由文本地理解析的准确度与消歧**——「哪个地名才是这条的落点」（多地名、机构名代地名、否定语境）决定信噪比。把每个实体都点上球会制造噪声，**与 D1/D3「低噪·宁静」直接冲突**；护城河在「选对唯一落点」的执行质量，而非把功能做出来。
3. **任意用户 feed 的取源约束**——NewsGlobe 是 Web，取任意 feed 要么服务端代理、要么受 CORS 限制（与 D14 网页/PWA 口径一致）。D14 把自定义 RSS 归**原生端（Capacitor 不受 CORS）**，配合 D16 零服务器，是相对 Web 竞品的一个**结构性取源优势**（无需自建代理即可读任意源）。
4. **信任与去重**——报道类源上球须过 D22 最小信任层（来源/时间/两级可信度、去重呈现），这是竞品普遍不做、也构成 Worlens 差异的一层，但同时是自身的工程前置成本。

**谁最可能先补齐（推测，非结论）：**
- **World Monitor** 概率最高：**已具条目级 geo + 开源 + 在活跃迭代**，从「curated feed」到「暴露用户加 feed 入口」是小步；短板同样是中文与「宁静低噪」呈现。
- **NewsGlobe** 已经做出核心读法，只差**非英文覆盖 + 移动端 + 打磨**；但作为无商业化信号的个人项目，能否推进不确定。
- **传统阅读器（Feedly/Inoreader/Readwise 等）最不可能**：与其「读源」定位相悖，且十余年未动。
- Worlens 的可守性**不能建立在「首创该读法」上**（已被 NewsGlobe 证伪），而应建立在：中文/多语种条目级解析 + 宁静低噪呈现 + 原生零服务器取源 + D22 信任层——即 market-202607 Q1 已给出的「执行层护城河」在本读法上的具体化。

## 具体用例回应：蓝点网（landiannews.com）单 feed 条目地理分散

### ✅ 事实
- 传统阅读器（Q1）只能把蓝点网整源归入一个文件夹，**无按条目地理拆分的选项**——用例描述与 Q1 核查一致。
- 已存在能对英文/欧语单 feed 做条目级地理落点的产品（NewsGlobe、World Monitor），但**未找到对中文 feed 做同类落点的在营产品**。来源见 Q3。

### 🔶 推测/判断
- 蓝点网用例恰好命中 Q4 门槛 #1（中文地理解析空档）：即便照搬竞品思路，「欧盟新闻/美国新闻各归其位」对**中文源**仍需自建中文地名解析——竞品现成方案（含 D13 T3 拟采用的 GDELT）以英文为主，对中文科技媒体覆盖存疑。此处既是最真实的差异化机会，也是最实的工程门槛。

## 遗留问题 / 未找到可靠信号

1. **NewsGlobe 功能为「站点自述」，未实测**：其「用户加 RSS + 条目级 geo」是官网声明，本轮未验证实际抽取质量、是否支持中文、是否长期在营。若该产品被列为直接对标，建议实测其对一条中文 feed（如蓝点网）的落点表现。标注：站点声明级，非实测。
2. **WorldLens Live / VoxOrb / The World Now 取源细节未穿透**：WorldLens Live 抓取 403，VoxOrb/The World Now 仅得营销摘要，「无用户加 feed」为**未找到证据**而非证否；如需硬结论建议实机核验。
3. **GeoRSS 老工具运行态未确证**：ACME GeoRSS Viewer / GeoNames converter 是否仍正常返回未逐一实访，仅据页面陈旧特征判为存档级。
4. **D13 T3（GDELT 地理编码）对中文源覆盖度未核**：GDELT 以英文为主，能否支撑蓝点网类中文科技 feed 的条目级落点，本轮未取得覆盖度数据——是决定蓝点网用例可行性的关键未知，建议列入下一轮或 arch 设计前置核查。
5. **付费信号缺失**：本读法相关产品（NewsGlobe 无定价、World Monitor Pro $39.99/月订阅）未给出「按地理读 feed」这一功能本身的付费验证；D14 升格为对外叙事若指望其驱动付费，仍属未验证假设。
