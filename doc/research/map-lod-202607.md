## 1. 分层惯例(Q1):随 zoom 分层显示国界与地名

### ✅ 事实

**A. Mapbox(矢量瓦片 + 碰撞避让,业界标杆实现)**
- **minzoom 逐要素/逐层控制**:zoom 取值 0–24,低于图层 minzoom 时该层隐藏。来源:[Mapbox zoom level 术语](https://docs.mapbox.com/help/glossary/zoom-level/)、[Style Spec Layers](https://docs.mapbox.com/style-spec/reference/layers/)。
- **碰撞检测(collision detection)**:一个瓦片内的标注集合通常大于屏幕不重叠可显示的集合,引擎自动挑选子集显示;`icon-allow-overlap`/`text-allow-overlap` 默认 false;两个标注冲突时**由要素绘制顺序(draw order)决定谁显示谁隐藏**;`variable label placement`(一个标注尝试多个候选位置)用于在保持可读的前提下提高标注密度。来源:[Collision Detection wiki](https://github.com/mapbox/mapbox-gl-native/wiki/Collision-Detection)、[Optimize map label placement](https://docs.mapbox.com/help/troubleshooting/optimize-map-label-placement/)、[Map Label Placement in Mapbox GL(博客)](https://blog.mapbox.com/map-label-placement-in-mapbox-gl-c6f843a7caaa)。
- **按 rank 抽稀(density thinning)**:官方「用 Natural Earth 优化地名标注」示例做法——用 `SCALERANK`(基于人口的重要度排名)作为排序字段;`lowest_where_in_distance` 把每个瓦片切成 64 个等距区块,**每个区块只保留 rank 最高的一个地名**;因瓦片数量随 zoom 呈二次增长,**保留的地名数量随 zoom 自然增加**,从而实现「远景只见首都/大城市 → 拉近才见中小城市」。来源:[Populated places — optimizing place labels(MTS 示例)](https://docs.mapbox.com/mapbox-tiling-service/examples/natural-earth-data-populated-places/)。

**B. Cesium(3D 地球,与本项目相机模型最接近)**
- **相机无「zoom level」概念,用相机距离**:`distanceDisplayCondition` 属性设定标注对相机的最小/最大可见距离,超出即隐藏——这正是「远景无国界/地名 → 近景出现」的直接机制。来源:[Cesium 社区:Labels based in distance/scale](https://community.cesium.com/t/labels-based-in-distance-scale/2434)、[End Point Dev: Cesium Labels](https://www.endpointdev.com/blog/2023/05/cesium-labels/)。
- **declutter(去杂)算法用 priority**:priority 用于最小化标注位移(静态标注保持固定,动态标注允许移动),相机/标注移动时标注在若干帧内平滑归位而非瞬跳。来源:[Cesium Declutter roadmap Issue #1097](https://github.com/CesiumGS/cesium/issues/1097)。
- **用 geometricError 做 LOD**:每个瓦片赋 `geometricError`,值越大表示应从越远处可见;拉近时 geometricError 大的瓦片标注先出现——「此法与 Google Earth 处理标注的方式一致(用矢量数据叠加响应 zoom 与相机距离,而非传统 2D 瓦片标注)」。来源:[End Point Dev: Place Labels via Vector Data Tiles](https://www.endpointdev.com/blog/2025/02/cesium-place-label-vector-data-tiles/)、[Migrating from Google Earth API](https://cesium.com/learn/cesiumjs-learn/cesiumjs-google-earth-api/cesiumjs-google-earth-api-part-ii/)。

**C. 通用注记密度控制惯例(跨厂商)**
- 核心三件套一致:**逐要素重要度 rank(scalerank/labelrank)+ zoom/相机距离门限 + 屏幕空间碰撞避让**;标注集合恒大于可显示集合,靠 rank 排序取子集。上述 A/B 两组来源互证。

**D.「地名随本地信息密度动态出现」的先例**
- **主流通用地图(Google/Apple/Mapbox 底图)未找到「因该地有新闻/事件才显示地名」的直接先例**——它们的标注显隐是**静态重要度 rank 驱动**(人口/行政级别),与「本地是否有实时信息」无关。检索了 Mapbox/Cesium/Google Earth 标注文档,均为 rank+zoom+碰撞模型。
- **最接近的可复用机制 = 数据驱动样式(data-driven styling)**:Mapbox 表达式(expressions)可按要素属性决定标注显隐——「可用条件逻辑依据要素是否具备某属性来决定标注文字」,官方示例即「机场标注依据 `ref` 属性是否存在来样式化」;`setFilter` 支持随状态变化动态过滤 symbol 图层。也就是说「某城市有 event → 打开其标注」这一逻辑在数据驱动样式里**有成熟机制先例**,只是通用底图默认不这么用。来源:[Apply conditional labels](https://docs.mapbox.com/studio-manual/examples/conditional-labels/)、[Filter symbols using a dynamic expression](https://docs.mapbox.com/mapbox-gl-js/example/filter-symbols-expression/)、[Expressions 参考](https://docs.mapbox.com/style-spec/reference/expressions/)。

### 🔶 推测/判断

- D24 想要的分层(远景无国界 → 中景国界 → 近景地名)**与业界惯例完全同构**,可直接照搬「rank 门限 + 相机距离 + 碰撞避让」三件套,不需要发明新范式。3D 场景下用「相机到球心距离」当 zoom 代理即可(同 Cesium `distanceDisplayCondition` 思路)。
- **「小城市仅当本地有事件才出现」是本产品的差异化点,而非现成惯例**:通用地图没有这个先例,但实现机制不新——等价于「给该城市标注一个 `hasEvent` 布尔,数据驱动显隐」。风险不在能否实现,而在**碰撞避让**:一旦某地突发大量事件、多个小城市标注同时点亮,仍需 rank/碰撞逻辑防止糊成一团。建议把「大城市固定 rank」与「事件点亮的小城市」纳入同一套碰撞优先级,而不是两套独立逻辑。
- 「大城市固定显示」的 rank 门限可直接取 Natural Earth 的 `scalerank`/`labelrank`(见第 2 节),省去自建重要度表。

## 2. 数据基础(Q2):Natural Earth 及同类公版数据

> 争议边界处置**移至第 3 节单独成节**,本节只覆盖许可、精度档位、体积、分级字段。

### ✅ 事实

**A. 许可(对零服务器打包 + 商用 + 上架最关键)**
- **全部为公有领域(public domain)**:「All versions of Natural Earth raster + vector map data found on this website are in the public domain.」**无需授权、无需署名**:「No permission is needed to use Natural Earth. Crediting the authors is unnecessary.」(可选署名「Made with Natural Earth」)。**免责声明**:作者「are not responsible for any problems relating to accuracy, content, design, and how it is used」。来源:[Natural Earth Terms of Use](https://www.naturalearthdata.com/about/terms-of-use/)。
- 公有领域含义(二手整理):无署名要求、无商用限制、无费用,是发布地图/Web/App 最易清权的选项。来源:[atlas.co: Natural Earth](https://atlas.co/data-sources/natural-earth/)。

**B. 精度档位与分级字段**
- 三档比例尺:**1:110m(粗)/ 1:50m(中)/ 1:10m(细)**,预综合(pre-generalized)由制图师人工筛选而非算法/众包生成,专为渲染优化。来源:[Natural Earth 首页](https://www.naturalearthdata.com/)。
- **admin-0 国界**:含 `scalerank`(原为印刷制图筛选设计)与 `labelrank`;**v4 系列新增 Web 地图优化字段 `min_zoom` / `min_label` / `max_label`**——可直接映射为「该要素在哪个 zoom/相机距离出现、标注在哪档显示」。来源:[Admin 0 – Countries](https://www.naturalearthdata.com/downloads/10m-cultural-vectors/10m-admin-0-countries/)、[v2.0.0 release notes](https://www.naturalearthdata.com/blog/miscellaneous/natural-earth-v2-0-0-release-notes/)。
- **国界线(admin-0 boundary lines)**:属性区分边界类型——**disputed / treaty / median line**(争议/条约/中间线);另有 200 海里界。来源:[Admin 0 – Boundary Lines](https://www.naturalearthdata.com/downloads/10m-cultural-vectors/10m-admin-0-boundary-lines/)。
- **人口聚居地(populated places)**:含全部 admin-0 与多数 admin-1 首府、主要城市/镇,加人烟稀少区的少量小镇取样;**排名以「区域重要性」优先于人口普查**;`scalerank` 供用户按需过滤地名数量;`PopMax` 列(考虑都市区总人口)用于**给地名标注定字号**;约 90% 城市有 LandScan 派生人口估值。三档 10m/50m/110m。来源:[Populated Places](https://www.naturalearthdata.com/downloads/10m-populated-places/)。

**C. 体积量级**
- 10m populated places 下载约 **540.67 KB**(完整列)/ **130.49 KB**(精简列)。来源:[Populated Places 页面](https://www.naturalearthdata.com/downloads/10m-populated-places/)。
- **50m/110m populated places 及各档 admin-0 边界的精确体积:未找到官网逐一列出的数字**(官网按档位分页,单页未给统一体积表)。量级判断见推测栏。

### 🔶 推测/判断

- **Natural Earth 是本项目边界/地名数据的默认答案**:公有领域 + 无署名 + 免责,完美契合「零服务器、随包打包、商用上架」的硬约束;`scalerank`/`labelrank`/`min_zoom` 三个字段直接喂给第 1 节的分层逻辑,几乎不用自建重要度表。
- **体积对移动端无压力(推断)**:矢量点/线数据本就轻,110m 档为「远景国界」用途,数据量远小于 10m;即便三档全打包,量级也在数百 KB~数 MB 级(10m populated places 完整版仅 ~540KB 为锚点),相较纹理贴图可忽略。建议**远景用 110m 边界、中近景切 50m、地名近景用 10m**,按相机距离切档兼顾体积与精度。
- 一个**必须在产品侧决策**的点:Natural Earth 的 `scalerank` 是「制图重要性」而非纯人口——某些区域小城因区域代表性被提级。若 D24 的「大城市固定显示」想严格按人口,需改用 `PopMax` 排序;若接受「制图重要性」则直接用 scalerank。此为口径选择,留给产品负责人/审美官。

## 3. 争议边界处置(Q2 硬风险点,单独成节)★

> **本节是上架 iOS 的硬风险点。以下为公开案例与常见做法梳理,非法律结论。任何据此的产品动作须走用户明示同意 + 法律审阅。**

### ✅ 事实

**A. 厂商对争议边界「因观看者所在地而不同」是既定行业做法**
- **Google Maps 按用户所在国重绘争议边界**:克什米尔在印度境内查看时作为印度一部分(实线),境外查看时以虚线标为「争议」;从巴基斯坦看则显示为争议。「From Argentina to the United Kingdom to Iran, the world's borders look different depending on where you're viewing them from.」Google 对争议区用灰色虚线「客观」呈现,并遵循各地本地法律。来源:[Washington Post: Google redraws disputed borders depending on who's looking](https://www.washingtonpost.com/technology/2020/02/14/google-maps-political-borders/)、[ORF: Drawing lines on the internet](https://www.orfonline.org/expert-speak/drawing-lines-internet-border-conflicts-politics-behind-digital-maps-51693)。
- **Apple Maps 亦按本地法律调整边界与地名标注**(responsive to local laws for border and place-name labelling)。来源:同上 ORF。
- **OpenStreetMap 用「on the ground(实地控制)」原则**,按实地情形绘制。来源:同上 ORF。

**B. Natural Earth 自身的争议边界政策(直接影响本产品打包哪套数据)**
- **默认按 de facto(实际控制)而非 de jure(法理)绘制**:「show political boundaries based on de facto status rather than de jure」。其后果——在默认 de facto 口径下,**台湾显示为独立、克里米亚归俄罗斯、巴勒斯坦归以色列**。这套口径「rigorous and self consistent」但不讨好所有人。来源:[Natural Earth Disputed Boundaries Policy](https://www.naturalearthdata.com/about/disputed-boundaries-policy/)。
- **v5(2021-12)新增多视角(multi-POV)支持**:官方支持 **31 个国家视角(Argentina、Bangladesh、Brazil、China、Egypt、France、Germany、Greece、India、Indonesia、Israel、Italy、Japan、Morocco、Nepal、Netherlands、Pakistan、Palestine、Poland、Portugal、Russia、Saudi Arabia、South Korea、Spain、Sweden、Taiwan、Turkey、Ukraine、United Kingdom、United States、Vietnam)+ ISO 国际标准**;争议要素用 `fclass_*` 属性(如 `fclass_iso`、`fclass_china`)承载不同视角的分类值,另提供「claim」线作为辅助下载。设计目标是「让教师无需担心法律合规就能教地理」。来源:同上 Disputed Boundaries Policy;[Admin 0 – Breakaway, Disputed Areas](https://www.naturalearthdata.com/downloads/10m-cultural-vectors/10m-admin-0-breakaway-disputed-areas/)。
- 即:**Natural Earth 已内置「切换视角」的数据机制**——理论上可为不同发行区打包不同 `fclass_*` 口径的边界。

**C. 中国区上架的合规现实(公开案例 + 法规)**
- **App Store 审查指南无「边界」专门编号条款**;相关约束落在 **Section 5(Legal)总则**:「Apps must comply with all legal requirements in any location where you make them available.」即合规义务下沉到当地法律。来源:[App Store Review Guidelines](https://developer.apple.com/app-store/review/guidelines/)(2026-07-21 抓取,正文未见边界专条)。
- **中国测绘法规(硬门槛)**:几乎每张在中国使用的地图都须经政府审核并取得**审图号(map approval number,如 GS(2024)xxxx)**;互联网地图服务须持 **IMS 许可**;**仅具备资质的中国境内出版者可申请地图审核,负面清单禁止外资投资地图出版业**;仅持 IMS 许可者不得自行采集/编制地图,须使用已获审图号的政府批准地图;在华提供在线地图服务须**将地图服务器置于中国境内**。来源:[Regulation on Digital Maps in China(中伦)](https://en.zhonglun.com/research/articles/52416.html)、[Navigating China's Digital Map Regulations(Lexology)](https://www.lexology.com/library/detail.aspx?g=9f755dba-12a6-47b5-8b47-c16200cb6d58)、[18 online map providers approved by China(TNW)](https://thenextweb.com/news/18-online-map-providers-approved-by-china-authorities-baidu-google-others-not-among-them-yet)。
- **中国地理数据限制(维基百科整理)**:中国境内地理信息使用受限于经国务院测绘主管部门特别授权的主体;并存在坐标偏移(GCJ-02)等技术性限制。来源:[Restrictions on geographic data in China(Wikipedia)](https://en.wikipedia.org/wiki/Restrictions_on_geographic_data_in_China)。
- **已知政治敏感内容下架案例**:Tech Transparency Project 发现中国区 App Store 缺失 16 个涉台应用(10 个新闻类、4 个涉台地理/餐饮);HKmap.live(香港抗议地图)多次被拒后,在《人民日报》评论质疑苹果次日被下架。来源:[TTP: Apple Is Censoring its App Store for China](https://www.techtransparencyproject.org/articles/apple-censoring-its-app-store-china)、[Michael Tsai: HKmap Live Rejected](https://mjtsai.com/blog/2019/10/03/hkmap-live-rejected-from-the-app-store/)、[Fortune: Apple removes HKmap.live](https://fortune.com/2019/10/10/apple-hong-kong-protest-map-hkmaplive-china-complaint/)。

### 🔶 推测/判断

- **这是 D24「计划上架 iOS」下最被低估的风险**:D24 把国界当作纯视觉/数据任务,但**边界画法在中国区是合规硬门槛,不是审美问题**。若直接打包 Natural Earth 默认 de facto 口径(台湾独立、藏南/克什米尔/南海按实际控制),**在中国区几乎必然触发合规问题乃至下架**。
- **可选项(仅陈列证据权重,不替产品拍板)**:
  1. **不做中国区**:仅上架海外区,用 Natural Earth 默认 de facto 口径——绕开中国测绘法,但放弃中国市场,且仍可能被个别国家(印度等)投诉。
  2. **按发行区切换 `fclass_*` 视角**:利用 Natural Earth v5 多视角数据,中国区打包 `fclass_china` 口径(九段线、藏南归中、台湾归中)。**但注意**:即便边界画对,中国测绘法要求的「审图号 + 境内服务器 + 资质出版者」三项本产品(零服务器、个人开发者、公版数据自绘)**基本无法满足**——合规不止于「画对线」。此路需专业法律确认可行性,权重上偏悲观。
  3. **中国区隐藏国界层**:D24 已定「国界默认随距离出现」,可让中国区发行版直接不加载国界矢量(只留海岸线+地名),规避边界画法争议;代价是中国区体验缩水。此为工程上最省事、风险最低的折中,但仍不解决「地名/台湾归属标注」等次级敏感点。
- **本产品自绘 + 公版数据的性质,决定它更像「电子地图/图示」而非「导航地图」**,理论上审图号适用边界存在解释空间;但**这属于必须由法律专业判断的灰区,RES 无法给结论**。强烈建议:上架前把「是否做中国区、做则如何满足测绘法」作为独立法律咨询事项,不要在工程排期里默认能过。
- 即便只做海外,**印度对藏南/克什米尔、阿根廷对马岛等仍可能投诉**;Natural Earth 默认 de facto 已是相对中立且「self-consistent」的选择,是海外发行的合理默认。

## 4. Apple Maps 风格可借鉴度(Q3)

### ✅ 事实

**A. 公开分析资料(权威来源)**
- **Justin O'Beirne** 是数字制图领域公认的深度分析者,长期逐帧对比 Apple/Google Maps 的制图差异(《Apple's New Map》《Google Maps's Moat》等);苹果高管曾受其分析触动而启动从零重建 Apple Maps。2020 年他记录 Apple Maps 数据更新(更清晰的绿地、地标细节等)。来源:[justinobeirne.com](https://www.justinobeirne.com/)、[pxlnv: An In-Depth Look at Apple's New Map Data](https://pxlnv.com/linklog/obeirne-new-apple-maps/)。
- **Apple 官方设计规范**:HIG Maps 页给出地图组件设计指引(供开发者)。来源:[Apple HIG: Maps](https://developer.apple.com/design/human-interface-guidelines/maps)。
- **Apple 制图风格具行业影响力的实证**:2020-08 Google Maps 开始测试「受 Apple Maps 启发」的样式——全球地表覆盖色更深更浓。来源:[Justin O'Beirne: Google Maps Testing Apple-Inspired Style](https://www.justinobeirne.com/google-maps-testing-new-apple-maps-inspired-map-style)。

**B. 可借鉴的通用制图设计语言要点(跨来源共识)**
- **配色**:限定调色板,最多约 12 种独立颜色(可有更多同色深浅);主元素(主干道、市中心)用**更深值 + 更高饱和度**,次元素(街区界、地形等高线)用**更浅值 + 更低饱和度**——即用明度/饱和度建立层次。来源:[Cartography Playground: Map design](https://cartography-playground.gitlab.io/playgrounds/map-design/)、[Map Library: 视觉层次](https://www.maplibrary.org/1483/visual-hierarchy-in-complex-map-designs/)。
- **注记字体与密度**:首都/主要地标/水体用较大字号(约 14–16pt),次级元素(街区、次要道路、POI)用较小字号(约 10–12pt);字体选择帮助用户快速区分地物类别并维持层次。来源:[Map Library: 排版](https://www.maplibrary.org/1591/enhancing-map-readability-through-typography-choices/)。

**C.「借鉴设计语言」vs「抄袭商业外观」的法律边界(trade dress)**
- **Trade dress(商业外观)是商标权的子集**,可保护网站/App 的「look and feel」;主张成立须满足三要件:**非功能性(non-functionality)、显著性(distinctiveness,产品设计须证明获得「第二含义 secondary meaning」)、混淆可能性(likelihood of confusion)**。来源:[Justia: Trade Dress](https://www.justia.com/intellectual-property/trademarks/trade-dress/)、[C&C IP: UI/UX 法律保护对比](https://www.candcip.com/single-post/legal-protection-of-user-interface-and-user-experience-design-comparing-trade-dress-copyright-and)。
- **法院普遍驳回宽泛的「look-and-feel」主张**:对概念性描述、结论性的非功能性主张、未具体证明「获得显著性」的诉求常予驳回;努力把「可保护的品牌识别」与「不可保护的、功能性的 UI 惯例」区分开。来源:[Lathrop GPM: Trade Dress Litigation in the Dupe Era](https://www.lathropgpm.com/insights/trade-dress-litigation-in-the-dupe-era-what-works-what-fails-and-early-court-signals/)。
- **Apple v. Samsung** 是基础性判例(设计专利在「整体视觉外观可能驱动购买决策」时保护广泛)。来源:[PatentPC: Famous Trade Dress Cases](https://patentpc.com/blog/famous-trade-dress-cases-that-changed-the-game-for-brand-owners)。
- **「因模仿地图风格引发纠纷」的具体先例:未找到**——检索 trade dress + 地图/制图样式模仿,未命中专门针对「地图配色/注记风格」判例;命中的均为通用 trade dress 与 Apple/Samsung 设计专利案。

### 🔶 推测/判断

- **可借鉴的是「设计语言/制图惯例」,不是「Apple 的具体外观」**:配色分明度/饱和度层次、首都大字号次级小字号、限定调色板——这些是**功能性、行业通行的制图惯例**,trade dress 一般不保护,借鉴风险低。真正有风险的是复制 Apple Maps**可识别的整体独特外观**(标志性配色组合 + 独特图标 + 整体版式)到「让用户误以为是 Apple 产品」的程度——那才可能落入 trade dress/设计专利。
- **对本产品的安全区(推断)**:学 Apple Maps「深色、低饱和、克制层次」的**方法论**,但用自己的配色数值与图形语言,避免 1:1 复刻其标志性色卡/图标。这与 D25「审美验收官」要落的「高级感」目标一致——目标是「同一审美流派」,不是「同一张脸」。
- 未找到地图风格模仿判例这件事本身是**弱正向信号**:说明制图设计语言的借鉴在业界属常态(Google 都公开「受 Apple 启发」),诉讼门槛高。但不构成法律背书,重大外观决策仍建议法律过目。

## 5. 深色矢量地球优秀参照(Q4)

### ✅ 事实

**A. cobe(Stripe/Vercel 风同款 5kB 点阵地球)**
- 作者 Shu Ding,库 [shuding/cobe](https://github.com/shuding/cobe),5kB WebGL。手法:**shader 画球体 + 球面斐波那契点阵(Spherical Fibonacci lattice)** 决定点分布(避免极点聚集);原用 4096×2048 世界地图纹理,**降采样到 256×128(~1KB)、base64 内联**,发现「点状世界地图看起来几乎一样」;叠加 **Phong 光照 + 辉光(glow)效果** 造就精致高级感。视觉核心 = **点阵 + 辉光 + 柔和光照**,全部 5kB。来源:[Shu Ding: COBE — WebGL Globe in 5kB](https://shud.in/thoughts/cobe)、[github.com/shuding/cobe](https://github.com/shuding/cobe)。

**B. GitHub 首页 globe(three.js,无纹理)**
- 官方工程博客拆解:跑在 three.js 的 WebGL 上;场景由 **5 层**组成——**halo(光晕)、globe(球体)、Earth's regions(陆地区域)、蓝色 spike(开放 PR)、粉色 arc(合并 PR)**;**完全不用纹理**——用 4 盏灯打在球上,用约 **12,000 个五边圆(five-sided circles)** 拼出陆地区域(嵌套经纬度循环、密度随机器性能可调),辉光用一个简单自定义 shader 画在球体背面。来源:[GitHub Blog: How we built the GitHub globe(2021-02-11)](https://github.blog/engineering/engineering-principles/how-we-built-the-github-globe/)。

**C. three-globe / globe.gl(vasturiano,开源 three.js 数据地球)**
- three-globe 是「ThreeJS WebGL 类,在球面投影上表现数据可视化层」,受 WebGL Globe 启发;globe.gl 是其便捷封装。支持图层:**points / arcs / polygons(可挤出锥/棱柱)/ paths / heatmaps / hex bins(集成 H3 六边形网格,分辨率 0–15)/ rings / labels / 自定义 3D 对象**;含 Country Polygons、Hexed Country Polygons、Labels 等现成示例。开源、活跃维护。来源:[globe.gl](https://globe.gl/)、[github.com/vasturiano/three-globe](https://github.com/vasturiano/three-globe)、[three-globe(npm)](https://www.npmjs.com/package/three-globe)。

**D. dataarts WebGL Globe(Google 早期开源范式)**
- Google Data Arts Team 的 [webgl-globe](https://github.com/dataarts/webgl-globe),经纬度数据可视化平台,是上述多数库的思想源头(three-globe 自述受其启发)。来源:同 three-globe 仓库自述。

**E. Flighty(dark 飞行地球,业界口碑高)**
- **未找到 Flighty 官方或第三方的详细技术/视觉拆解**:检索「Flighty app globe design」仅返回泛化的第三方 flight-globe 项目(FlightGlobe、Flight Viz、YourFlights 的 night earth 视图等),**非 Flighty 本体的配色/发光手法拆解**。标注:未找到可靠公开拆解。若需参照,建议实机截图由审美官(D25)做视觉分析,而非引用不存在的公开文档。

### 🔶 推测/判断

- **两条高级感技术路线,本项目二选一或融合**:
  1. **cobe 式点阵球**:5kB、无重纹理、点阵+辉光,极简高级——但点阵世界地图与「精确国界线/地名」是两种语言,若要 D24 的清晰国界+可读地名,点阵更适合做**底球质感**而非承载精确边界。
  2. **GitHub globe / three-globe 式矢量层**:无纹理、多层叠加(球体 + 区域 + 线 + 标注),**天然契合 D24「海岸线 + 国界线 + 地名」的分层诉求**;three-globe 的 polygons/labels 图层几乎是 D24 需求的现成拼图。
- **本项目现状(three.js 自绘海岸线+经纬网)与 three-globe/GitHub globe 是同一技术家族**,借鉴门槛最低;cobe 更多是「审美灵感 + 辉光/光照手法」的参照(尤其昼夜半球柔和明暗对比,可借鉴其 Phong 光照做法)。
- **D24「昼夜半球柔和明暗对比」的直接技术参照 = 定向光 + Phong/柔和光照**(cobe、GitHub globe 都是「打灯到球上」而非贴纹理),这正是把「整球均匀深色」改成「有半球级明暗」的最简路径——调一盏方向光(代表太阳)即得半球对比,晨昏线锐利度由光照过渡宽度控制。

## 6. 可行性扫描(Q5):three.js 自绘分级注记/边界线常见路径

> 仅到「常见方案」层面,不做技术设计(技术设计属 arch)。

### ✅ 事实(均有开源实现佐证)

- **国界线绘制**:加载 Natural Earth admin-0 的 GeoJSON,经纬度 → 球面 3D 坐标后用 `THREE.Line`/线几何绘制;three-globe 的 **polygons/paths 图层**即做此事(GeoJSON 定义区域挤出或描边)。来源:[globe.gl 示例(Country Polygons)](https://globe.gl/)、[three-globe](https://github.com/vasturiano/three-globe)。
- **分级注记(labels)**:three-globe/globe.gl 有 **labels 图层**;另一常见路径是用 three.js `CSS2DRenderer`/sprite 把 HTML 文本贴到球面点,**按相机距离控制显隐**(等价 Cesium `distanceDisplayCondition`)。来源:同上;[Cesium distanceDisplayCondition 讨论](https://community.cesium.com/t/labels-based-in-distance-scale/2434)(机制参照)。
- **LOD 切档**:按 `camera.position.length()`(相机到球心距离)设阈值,切换 Natural Earth 110m→50m→10m 数据或切换图层可见性;逐要素用 `scalerank`/`min_zoom` 决定「相机距离低于某值才显示该边界/地名」。来源:Natural Earth 字段(见第 2 节)+ Cesium/Mapbox rank 惯例(第 1 节)。
- **碰撞避让(已知难点)**:sprite/HTML 标注的碰撞避让在 three.js 里**需自行做屏幕空间重叠检测**,three-globe 不提供完整碰撞引擎;Mapbox-gl 有成熟碰撞但它是整套瓦片引擎、非纯 three.js。这是自绘路线的已知短板。来源:[Mapbox Collision Detection](https://github.com/mapbox/mapbox-gl-native/wiki/Collision-Detection)(对比参照)、three-globe 文档(无碰撞层)。
- **开源可抄范例**:[three-globe 示例](https://github.com/vasturiano/three-globe)(country-polygons / labels / hex)、[globe.gl](https://globe.gl/)、[janarosmonaliev/github-globe](https://github.com/janarosmonaliev/github-globe)(GitHub globe 复刻)、[shuding/cobe](https://github.com/shuding/cobe)、[dataarts/webgl-globe](https://github.com/dataarts/webgl-globe)。

### 🔶 推测/判断

- **技术路径无拦路虎,唯一真难点是标注碰撞避让**:边界线、按距离切档、按 rank 显隐地名都有现成开源做法;但「大城市固定 + 小城市事件点亮 + 不糊成一团」需要自建屏幕空间碰撞/优先级逻辑,three-globe 不给,得自己写或裁剪 Mapbox 思路。建议 arch 阶段把「标注碰撞」单列为风险项。
- **最省力方案(推断)**:直接引入 three-globe 作为图层库(polygons 画国界、labels 画地名),而非从零手写——它与本项目 three.js 栈同源、开源、覆盖 D24 绝大多数图层需求;仅碰撞与「事件驱动显隐」需在其上定制。是否引入第三方库属工程取舍,留给 arch/orch。
- cobe 的点阵球与 D24 的「精确国界+地名」诉求语言不同,更适合当**底球质感/光照参照**;不建议用点阵承载国界。

## 综合判断(推测栏,供产品负责人参考)

1. **D24 的分层 LOD 与业界完全同构、Natural Earth 完美满足零服务器公版约束——技术与数据侧无实质障碍**,可直接落地。
2. **「地名随本地事件动态出现」是差异化点、无通用地图先例,但机制不新**(数据驱动显隐),真难点是并发点亮时的碰撞避让。
3. **最大且被低估的风险在争议边界的 iOS 上架合规(第 3 节)**——这不是审美问题而是法律/合规硬门槛,尤其中国区涉及审图号/境内服务器/资质出版者三重门,本产品的零服务器+个人开发者+公版自绘性质基本撞墙;需产品负责人拍板「是否做中国区」并做独立法律咨询,不能在工程排期里默认能过。
4. **借鉴 Apple Maps「深色低饱和克制层次」的设计语言风险低**(制图惯例、无地图风格模仿判例),但避免 1:1 复刻其标志性外观;与 D25 审美官目标一致。
5. **深色高级感的现成参照按同源度排序**:three-globe/GitHub globe(同 three.js 矢量家族,首选借鉴)> cobe(辉光/光照/昼夜对比手法参照);Flighty 缺公开拆解,建议实机截图交审美官分析。
