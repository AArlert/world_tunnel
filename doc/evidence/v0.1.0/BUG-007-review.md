## 逐条核对结论

### 1. 三条规则文本与 doc/bugs.md 登记期望的一致性

逐字比对现文本与 bugs.md「**修复**」段落列出的 ①②③：

- ①「写共享台账的卡一律串行，判断依据是『这张卡是否可能写该文件』而非『回填目标是否不同』」
  ——SKILL.md 现文本：「判断依据只有一条——这张卡是否可能写该文件，按卡的实际写入面判断，
  不按角色、不按『回填目标是否不同』」。一致。
- ②「改 scripts/ 的卡与用 scripts/ 的卡不并行（避免 qa 跑 evidence 时拿到中间状态）」——SKILL.md
  现文本原文照录，一致。
- ③「单卡范围上限约 2 个场景或 2 条缺陷，超出即拆（5 场景卡曾整卡空转 55 分钟无产出，拆为 2
  场景后 3.6 分钟完成）」——SKILL.md 现文本原文照录，一致。

结论：**一致，通过**（与首轮复验的该项结论相同，本轮复核未见出入）。

### 2. 首轮 FAIL 指出的漏洞（"只读该文件的卡（如 rev）不受此限"举例）是否已消除

首轮复验指出：规则①末尾曾新增举例「只读该文件的卡（如 rev）不受此限」，与「按卡实际写入面
判断、不按角色」的原则矛盾——因为 rev 做**仲裁**类任务时依 rev.md 的权限列表（Edit/Write 权限
含「bugs.md 的裁决与状态列」）确实要写 doc/bugs.md，只有门禁/审查/签核三类任务才是只读。

核对现文本：该原「如 rev 不受此限」的字面表述**已不存在**于规则①正文中。现文本改为：

> 「rev 做**仲裁**时要写 bugs.md 的裁决与状态列（rev 只有门禁/审查/签核三类任务才是只读），
> 都会落到 bugs.md。**不给任何角色开无条件豁免**——BUG-007 首轮修复就栽在『如 rev 不受此限』
> 这句举例上，把判断依据偷换回了角色刻板印象。」

比对 rev.md 原文核实：「Edit/Write 权限仅用于：审查/签核记录（doc/review/、doc/evidence/ 下）、
bugs.md 的裁决与状态列、doc/bugs/<BUG-ID>.md 的仲裁结论段」，且 rev 的四类任务中仅「代码/测试
审查」「里程碑签核」「arch 交付门禁」三类不写 bugs.md，「仲裁」一类明确要写。现文本的表述与
rev.md 原文逐条对得上，未见曲解。

本次修复采用的是首轮建议的方向 a（按 rev 四类任务区分，仲裁纳入串行范围），同时保留了对首轮
错误举例的历史说明（未采用方向 b 的直接删除，但效果等价——错误表述已不再作为现行规则出现，
仅作反面案例引用，行文清楚标注为「首轮修复就栽在」，不会被误读为现行豁免条款）。

结论：**该具体漏洞已消除，通过**。

### 3. 排查项：arch 的 spec 提案卡是否也可能写 doc/bugs.md，是否已被现规则覆盖

核对 `.claude/agents/arch.md`：

- 职责段明确写明：「spec 修改提案：发现歧义或需要工程适配时给出具体改法（原文/新文/理由/
  影响面），**登记 bugs.md** 或附在交付里……」——即 arch 在做 spec 修改提案类任务时，「登记
  bugs.md」是明确列出的两个选项之一，**确实可能直接写 doc/bugs.md**。
- arch.md 的工具声明为 `tools: Read, Grep, Glob, Edit, Write, Bash`（无路径限制），且全文**没有**
  类似 rev.md「禁区」段那样显式限定 Edit/Write 可写文件范围的条款——即 arch 对 doc/bugs.md 的
  写入没有被文档层面禁止或限制。
- 结论：arch 的 spec 提案卡**确实可能写 doc/bugs.md**，与 qa/dev/rev 三者同属需要纳入串行判断
  的角色，首轮遗留的排查建议属实。

现规则文本是否已覆盖此项：

- 规则①开篇即给出角色无关的通用判据：「判断依据只有一条——这张卡是否可能写该文件，按卡的
  实际写入面判断，**不按角色**」；段末再次强调「**不给任何角色开无条件豁免**」。这两处表述
  均为无角色限定的强通用语句，逐字检查未发现任何字面上把 arch 排除在外、或暗示"仅
  qa/dev/rev 三类角色需要纳入判断"的措辞——qa/dev/rev 三例是作为「都会落到 bugs.md」的例证
  列出（"都会"一词表明是举例而非穷举式限定），并非对适用范围的排他性声明。
- 与首轮判定为 FAIL 的情形对比：首轮问题的性质是**文本中出现了一句明确的、错误的豁免声明**
  （"如 rev 不受此限"），直接授权了错误行为；本轮排查 arch 的情形是**规则示例未显式点名
  arch**，但没有任何一句现行文本明确或暗示 arch 被豁免。两者性质不同：前者是"肯定式错误
  断言"，后者是"举例未穷举"。按同一标准衡量，只有当规则文本本身包含误导性/错误的断言时才
  构成同类漏洞；单纯的举例未穷举，在通用判据已经明确"不按角色""不给任何角色开无条件豁免"
  的前提下，不构成对现行规则有效性的实质破坏。
- 结论：**该项已被现规则的通用判据在字面上覆盖，判定通过**；但存在下述遗留风险，建议后续
  加固（非阻塞）。
