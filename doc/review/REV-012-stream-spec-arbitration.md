## 1. proposal-stream-order（BUG-024）逐项裁决

### 1.1 根因与方案主线：**准**

- 现行 SPEC-2.2a 排序句「列表按 `ts` 倒序」是在「事件流＝纯过去时间轴」隐含前提下写的，未考虑 SPEC-5.5 的 LL2 `ts=net`（未来时刻）。这是**规格级设计缺陷**（对外可见行为定义错误），走 §7 成立，非实现 bug。
- 方案 1.1（`|ts − now|` 升序，双向向 now 收敛）满足 BUG-024 判据「首屏＝最紧迫」，且三边界（空态/全未来/全过去）用同一规则退化覆盖：全过去退化为 `ts` 倒序（与现行等价、零回归），全未来退化为倒计时序。**与 D23 L0「多新」编码意图（行 72）一致**。方案 A/B/C/D/E 的否决理由（提案 §1.2）经核，成立——尤其否决 D（severity 排序）避免与 BUG-025 卡耦合，边界清晰。**准**。

### 1.2 tie-break（`|ts−now|` 相等时未来先于过去、再 `id` 升序）：**准**

- 该并列消解为**测度零场景**（精确等距），其唯一硬要求是**确定性**（供 QA 断言）。`id` 为 `{source}:{原始id}` 全局唯一（SPEC-6.1，行 127），`id` 升序是全序 → 完全确定。
- `ts=now` 归过去侧（`ts≤now`）已明确，不产生 NaN/抖动（提案 §2「恰在 now」行）。**准**。
- QA 注记：排序类断言须 mock `now` 于固定瞬时并构造相对 `ts`，等距用例才可复现确定断言（`now` 为移动基准，仅在评估瞬时定序）。

### 1.3 §7 遗留项①「severity 是否应为更高优先级排序键」：**当场裁定关闭——severity 不作排序键**

- 提案将此列为「跨两卡联动议题」并**回避拍板**。因两卡在本仲裁**同时在案**，rev 有条件当场裁定，避免悬空：
  - **裁定：时间邻近度为列表唯一排序轴；severity 不进入排序键。** severity 的注意力分配由**行视觉轻重编码**（BUG-025，proposal §2）承担，与排序**正交**——这正是「排序＝时间邻近（多新/紧迫）、编码＝severity（多重）」的干净分离。
  - 依据：若 severity 作排序主键，会使「3 天前的 severity-3」压过「2 分钟前的 severity-1」，直接违背 D23 L0「多新」（行 72）与「事件流」从属语义（D3，行 14）；且 severity 仅 3 档、档内仍需时间定序，作主键并不解决任何问题、只制造两轴纠缠。
  - 该裁定**关闭** proposal-stream-order §7 项① 与 proposal-event-noise-severity §1.4 的联动悬置：未来若产品仍想「高 severity 置顶」，须另起 §7 提案单独论证，不得默认预埋。

### 1.4 §7 遗留项②「重排视觉 churn」：**准（非 spec）**

- `now` 推进导致未来发射向顶攀升、跨 T-0 后转过去下漂，属预期行为且与相对时间刷新同源。若高频刷新出现视觉抖动，属**实现层（刷新节流）**问题，登记 DEV 侧优化，**不改排序语义**。**准**。

### 1.5 SPEC-2.2a 排序句文本：**逐字采纳**（并入 §5.3 合并稿，无改）

## 2. proposal-event-noise-severity —— BUG-023（呈现门槛）逐项裁决

### 2.1 入球/入流统一为单一呈现集：**准**

- 拆分入球/入流门槛将破坏 M2-13 现契约「列表条目数 ↔ 球面标记数一一对应」（testplan 行 37），且 BUG-023 痛点同时是「球面 200+ 红点」与「事件流被淹没」，单侧门槛不闭合缺陷。D3「球为本体、列表从属」（行 14）要求二者呈现同一事实集。**准**。

### 2.2 §7 项①「USGS 源侧换 feed vs 客户端过滤」：**准源侧换 feed**

- 裁定采纳**源侧换 `2.5_hour`/`2.5_day` 显著性 feed**，否决「保留 `all_*` + 客户端过滤」。理由：
  1. **SPEC-3.10 性能预算**（行 73，首包 ≤2MB / 上屏 ≤1s）：`2.5_day` 载荷约为 `all_day` 十分之一量级；每 60s 下载 250+ 条再丢弃逾九成是净退步。
  2. **极简**（CLAUDE.md §1.2）：零新增过滤代码，门槛即 feed 边界，`severityFromMag` 一字不改。
  3. **门槛值可辩护**：M≥2.5 是 USGS 自身发布的显著性 feed 分级，比自拟阈值更利 QA 锚定。
- **客户端 `mag<2.5` 兜底**：**不写进 spec**（保持 spec 最省）；作为 DEV 实现自由度允许。spec 对外契约 = 「USGS 门槛 = M≥2.5，源侧 feed 兑现」。**准**。

### 2.3 §7 项②「门槛值 M≥2.5 vs M≥3.0」：**准 M≥2.5**

- 裁定取 **M≥2.5**，否决上调至 3.0。理由：
  1. USGS 有 `2.5` 现成 feed；3.0 无现成 feed，须客户端过滤（丢失零代码收益）或退到 `4.5` feed——而 `4.5` feed 会丢弃**全部 severity-1 地震**（SPEC-5.1 `mag<4.5→1`，行 92），使 USGS severity-1 分支成为死区，过激。
  2. M2.5–4.5 为有感小震（部分可感），保留符合 D22「少而准」而非「少而失真」。
- **准 M≥2.5**。

### 2.4 §7 项③「GDACS Green 是否需门槛」：**准不设门槛**

- 裁定 GDACS/EONET/LL2 **均不设门槛**。理由：无实证噪声证据（GDACS 源侧已按 `alertlevel` 甄别、量小；EONET fixture 26 条；LL2 `limit=10`）；SPEC-5.0a 通则要求门槛「须以实证噪声为据」，无据设门槛违反极简（CLAUDE.md §1.2）。若未来实测 Green 为噪声，另经 §7 加门槛。**准不设**。

### 2.5 三机制分工（门槛 / severity 编码 / 聚合）：**准**

- 提案 §1.4 表经核：门槛（全体归一化事件、二值 in/out）、severity 编码（已准入集、连续视觉轻重）、聚合（已准入且地理密集、空间合并）作用于不同群体不同轴，**不重叠、无重复解决**（M0.35 只被门槛处理、永不到达编码/聚合）。**准**。

### 5.1 新增 SPEC-5.0a（紧随 SPEC-5.0，§5 跨源统一约束）

> - **SPEC-5.0a 呈现门槛（surfacing floor，跨源统一约束）**：每源可声明一个**显著性门槛**；仅达到/高于门槛的归一化事件进入**呈现集**——同时决定球面标记（§3）与事件流列表（SPEC-2.2/2.2a）是否出现该事件，**球面与列表共用同一呈现集**（维持 SPEC-2.2「事件流」对球面的从属语义）。低于门槛的事件**既不入球也不入流**。门槛以最省的可用手段实现——源侧提供预筛（feed/查询参数）时优先源侧（见 SPEC-5.1），否则客户端谓词过滤；**实现位置属实现自由度**，对外契约仅为「亚门槛事件不呈现」。**无噪声问题的源不声明门槛（呈现全部事件）**：EONET/GDACS/LL2 现均无门槛。新增或变更任一源的门槛属 spec 变更（§7），须以**实证噪声**为据。呈现门槛只做**准入**（二值 in/out），与密集区**聚合**（D23 L0）、列表行 severity **轻重编码**（SPEC-2.2a）分属不同机制、互不替代。

> - **SPEC-2.2a** 事件流面板列表行：SPEC-2.2 事件流面板的每个列表行显示三项——① 分类色圆点（取 SPEC-3.7 对应 category 色）；② 事件标题（SPEC-6.1 `title`）；③ 相对时间（由 SPEC-6.1 `ts` 相对当前时刻派生，如「3 分钟前」，格式细节属实现自由度）。**行的整体视觉轻重编码事件 severity（SPEC-6.1，severity∈{1,2,3}）**：跨三档**单调递增**——severity 1 最轻（弱化、注意力最低）、severity 3 最重（最突出）；编码**仅用非色相通道**（尺寸/字重/透明度/行密度等），**色相仅表分类（SPEC-3.7），severity 不引入任何新增色语义**（与 marker 侧 SPEC-3.7「尺寸/脉冲随级递增」同原则，避免 BUG-022 式第二色语义）；**具体承载通道与其取值由 D25 审美官方案定（待 D25 定值）**。**列表按「距当前时刻 `now` 的时间邻近度」升序排序（`now` 与③相对时间同源、随之刷新）：主排序键为 `|ts − now|`，值越小（离当前时刻越近——无论刚发生的过去事件，还是临近发生的未来事件如发射倒计时）越靠前，值越大（越久远的过去或越遥远的未来）越靠后；`|ts − now|` 相等时未来事件（`ts > now`）先于过去事件（`ts ≤ now`），仍并列者按 `id` 升序稳定排序以保确定性。全部为过去事件时该规则等价于 `ts` 倒序（最新在上，保留「事件流」直觉）；全部为未来事件时等价于最近发生者在上（倒计时序）。`ts` 仅供展示/排序、不参与过期判定（SPEC-6.3①）。** 无事件时显示空状态文案「暂无事件」。severity 的**数值/文字/徽章不作为显式行元素**（其数值展示属详情卡 SPEC-2.3，M3）——行内仅以上述视觉轻重环境式编码 severity 档位，不新增可读元素；地点、摘要、信源等完整字段属详情卡（SPEC-2.3，M3），不进列表行。列表所呈现的事件集受 SPEC-5.0a 呈现门槛约束。
