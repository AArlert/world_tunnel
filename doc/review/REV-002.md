## 2. 设计与 spec 的一致性

- **SPEC-6.2 / SPEC-3.6 对齐推导：核实通过。** 已按 `src/globe/geo.ts:9` 的实现（`x=cosφ·sinλ, y=sinφ, z=cosφ·cosλ`）验算 §4.1 的两种修法：
  - 方案 1 `phiStart=-π/2`：u=0.5 → φ=π/2 → 顶点落 +Z（= `latLonToVector3(0,0)`）；u=0.75 → φ=π → 顶点落 +X（= `latLonToVector3(0,90)`），u 向东递增。与 SPEC-6.2「(0,0)→+Z；(0,90°E)→+X」一致，**推导成立**。
  - 方案 2 `phiStart=0 + offset.x=0.25 + RepeatWrapping`：+Z 处 u=0.25，采样坐标 0.5，等价成立。两方案确为等价，给 DEV 二选一合理。
  - 竖直方向：thetaStart=0 在 +Y、uv.y=1 对图片顶行，与 SPEC-6.2 北极→+Y 一致。
  - §4.1 末段禁用「片元内 `atan(N.x,N.z)` 解析算 uv」的理由（接缝 mip 跳变）成立，且该写法会绕开 `createEarthGeometry()` 这一「uv 约定唯一出处」，使检查点 6/7 的单测失效 —— 该禁令**应保留为硬约束**，建议 arch 把它从 §4「实现提示（不构成强约束）」上移到 §2.2。
- **SPEC-4.5 数据流**：§3.4 的 `subsolarPoint → latLonToVector3 → uSunDir(模型空间)` 与 SPEC-4.5「经 SPEC-6.2 坐标约定转成单位向量」逐字一致。「uSunDir 不随 earthGroup.rotation.y 变换、shader 用模型空间法线」是 SPEC-4.5 + SPEC-3.5 + SPEC-7.3 的必然推论（否则自转会拖动晨昏线），**自洽**。

### 5.1 P1 — 年积日 N 的起点定义 → **采纳（原案措辞小幅收紧）**

**依据（独立复算，非采信 arch 转述）**：对 `δ = 23.44·sin(2π(284+N)/365)` 逐点计算：

| 锚点 | Jan1=1 | Jan1=0 | SPEC-4.4 容差 ±1° |
| --- | --- | --- | --- |
| 3-20 平年 | N=79 → δ=**−0.807°** | N=78 → δ=**−1.210°** | Jan1=0 **FAIL** |
| 3-20 闰年 | N=80 → δ=−0.403° | N=79 → δ=−0.807° | 均 PASS |
| 6-21 | N=172 → δ=+23.440° | N=171 → δ=+23.435° | 均 PASS |
| 12-21 | N=355 → δ=−23.440° | N=354 → δ=−23.438° | 均 PASS |

「1 月 1 日 = 0」在平年春分锚点上超出 SPEC-4.4 容差，故 SPEC-4.4 已隐含唯一解，P1 只是把隐含约定显式化，不改变任何既有判定 —— 属最小改动。同时复算 EoT 全年区间为 **[−14.60, +16.45] 分钟**，在 Jan1=1 约定下满足 SPEC-4.4 的 |EoT| ≤ 17；EoT≈0 日期取 4-15（N=105）时 UTC 正午直下点经度 = **+0.056°**，满足 ≈0°。即三项 4.4 锚点在本裁决下全部自洽。

**建议新文（SPEC-4.1 末尾追加）**：
> N 为年积日，按 **UTC 日期**计，1 月 1 日 N=1；闰年 2 月 29 日照常顺次计入（公式分母恒为 365，不因闰年调整）。SPEC-4.2 的 N 取同一定义。

**影响面**：testplan M1-01 / M1-02 的期望值获得锚点；design-prompt §2.1 末条与 §3.2 的 `dayOfYearUTC` 由「临时约定」转为 spec 引用。

### 5.4 P4 — 拖拽/自转的作用对象 → **采纳（措辞合并到 SPEC-7.1 与 7.3）**

**依据**：三条现有条目共同蕴含该结论但均无明文——SPEC-3.5「星空随相机旋转（不随地球自转）」要求「相机旋转」与「地球自转」是两个不同的自由度，否则该括号无意义；SPEC-7.1「垂直视角限制在**纬度** ±85°」以相机纬度为量纲，指向相机运动；SPEC-7.3 则称「自转」。按 CLAUDE.md §7，QA 断言只准从 spec 推导并引用条目号，当前无条可引，属必须消解的歧义。同时本裁决直接修正 testplan T-1 的矛盾。

**建议新文**：
- SPEC-7.1 追加：「拖拽与其后的惯性作用于**相机**（绕球心的方位角/仰角），地球本体不因拖拽转动。」
- SPEC-7.3 追加：「空闲自转作用于**地球本体**（绕地轴，即 SPEC-6.2 的 +Y 轴），相机与星空不动；自转期间晨昏线与地理位置的相对关系保持不变（晨昏线由 SPEC-4.5 的真实时刻驱动，不随自转移动）。」

末半句是 SPEC-4.5 + 本条的必要推论，写明可防止 DEV 把 sunDir 放进世界空间——这正是 design-prompt §3.4 反复强调的最易错点，写进 spec 后 QA 才能对其断言。

**影响面**：testplan M1-09（须按 T-1 修订）、M1-11、M1-08；design-prompt §2.3 由「联合推导」升格为 spec 引用。

### 5.5 P5 — 纹理未就绪期的可见表现 → **修改后采纳**

**依据**：SPEC-3.2 定义了纹理就绪后的呈现，未定义就绪前。5400×2700 JPEG 的解码延迟使该窗口稳定可见，属真实的对外可见行为，按 §7 必须进 spec 而不能留给 DEV。裁决取「最小可见面」方向：新增 UI（loading 文案/进度条/骨架屏）会牵出文案、i18n、位置、退出动画一连串未定义行为，违反 §1.2；纯色占位是零新增语义的最小解。

**建议新文（SPEC-3.2 末尾追加）**：
> 纹理就绪前：球体以深色纯色（近夜面色）渲染，**不显示**加载文案、进度指示或骨架动画；纹理加载完成后直接替换，无淡入过渡。加载失败不重试、不弹错误 UI，保持占位渲染（错误细节仅进 console）。

「加载失败」一句为 rev 追加：P5 只问了「未就绪」，但失败是同一窗口的必然分支，不写会在 M1 留下第二个未定义可见行为。若产品负责人希望失败时有可见提示，属产品决策，请 orch 回带确认——rev 按最小可见面裁决。

**影响面**：design-prompt §4.5 由临时约定转为 spec 引用（见 C3）；不新增 testplan 场景（该行为无稳定可复现的自动化判据，靠 code review 保证），如需覆盖可由 qa 在 M1-05 附带截图说明。
