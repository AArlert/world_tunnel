## 1. 交付门禁逐项结论

### 1.1 门禁 ① 无子句静默蒸发：**不成立**

独立枚举现行 spec 全部子句（SPEC-1、2.1–2.4、3.1–3.8、4.1–4.5、5.0–5.7、6.1–6.3、7.1–7.5、8.1–8.5、§9），逐条比对新划分归属。绝大多数有正确承接（详见 §1.5 与提案 §3.2），**但两处「done」标记与 M1 实际交付不符，构成静默蒸发**：

**K-1（主打回项）：SPEC-7.4 被误标「done(FM-04)」，实际 M1「不做」且无 FM 承接。**

- 提案 §3.2 line 298 写「SPEC-7.1~7.5 done(FM-04)」。
- 独立回验：`doc/design-prompt/M1-globe.md:13` 明确将「点击飞行与详情卡（SPEC-7.4）」列入 **不做**；line 227「SPEC-7.4 的相机飞行是 M2/M3 的事……不要预留 flyTo 空壳」。M1 testplan M1-09/10/11（FM-04）只覆盖拖拽/缩放/惯性/自转（SPEC-7.1/7.2/7.3/7.5），**无任何点击飞行、详情卡、列表联动**。REV-004 §5 R-3 已明确 SPEC-7.4「无 testplan 场景……须显式建行」。
- 后果：SPEC-7.4（点击标记/列表条目→相机 800ms 飞行 + 弹详情卡 + 列表联动）在新表**无任何 FM 行锚点**（FM-07 锚点为 SPEC-2.2/3.7/3.8；FM-14 锚点为 SPEC-2.3/2.1），且 §3.2 将其误判为已完成。这正是 REV-002 G-1 / BUG-014 警示的失效模式：子句因被判「done」而永不再被任一里程碑「场景全 ✅」提示，静默蒸发。
- 整改：§3.2 更正 SPEC-7.4 为未交付；将 SPEC-7.4 拆挂 FM-07（列表 hover/选中↔标记高亮联动，随标记层 M2）与 FM-14（点击飞行 800ms + 弹详情卡，随详情卡 M3）；§4 相应补 M2/M3 testplan 覆盖点（点击飞行、列表↔标记双向联动）。

**K-2：SPEC-3.4「不遮挡标记」M2 回补义务被「done(FM-03)」掩盖。**

- 提案 §3.2 写「3.4/3.5/3.6 done(FM-03)」。

- 后果：SPEC-3.4 的「不遮挡标记」维度尚未真实验证，标「done」会使该维度无 M2 回补场景兜底。注意提案对 SPEC-3.8 处理正确（§3.2「3.7/3.8 FM-07/FM-11」，未标 done），唯 SPEC-3.4 漏网。

## 2. 歧义仲裁（逐条书面裁决）

### 2.1 A1 — 开屏锚定（D8/M3）依赖地点 watchlist（D6/M4）的时序冲突：**方向裁准，条件整改（K-4）**

- 裁决：arch 的分层方向成立——M3 开屏锚定的区域来源改由首启引导（SPEC-8.6）提供、M4 的地点圆域 watchlist 再精化，避免 D8 完整形态被迫整体后移 M4。SPEC-3.1(2.4) 已含安全兜底「未选定时沿用默认」，故不存在硬矛盾。
- **但发现一处内部不自洽（K-4，须整改后方能落地该分层）**：SPEC-8.6(2.18) 现文把首启引导定性为**垂类分流入口（冲突观察/航天/灾害等）**，即按**主题类别**采集偏好；而 A1 的解法依赖引导提供**粗粒度地理区域**。类别选择（如「航天」）不产出经纬度区域，无法驱动开屏锚定。若不修，M3 的引导只采集垂类、地点偏好要等 M4，则 SPEC-3.1 的锚定在 M3 永远命中「未选定→默认」分支，D8 在 M3 空转、实质滑到 M4。
- 整改要求：SPEC-8.6 正文须显式增加「粗粒度区域/关注地」采集维度（与垂类分流并列），使开屏锚定在 M3 有真实数据源；SPEC-3.1(2.4) 的「首启引导……选定关注区域」措辞方能成立。

### 2.2 A3 — SPEC-3.2 重写后 M1-05/M1-06/M1-14 已 ✅ 场景的复核口径：**保留 ✅ 予以裁准（rev 书面裁决）**

依 §5.2「✅ 撤销须 rev 书面裁决」，本审查员就此出具裁决：

- 独立复核被测行为在重写后是否仍为真：M1-05 验 SPEC-3.6 纹理经度对齐（几内亚湾校准）；M1-06/M1-14 验 SPEC-3.3 夜景增益 ≥1.5、昼半球不叠加灯光、昼夜为混合而非叠加。SPEC-3.2(2.5)/3.3(2.6) 重写将「昼夜/晨昏线」与「地表底图风格」**解耦**，明确昼夜混合、晨昏线过渡带 t∈[-0.1,+0.1]、「昼半球不叠加灯光、昼夜为混合而非叠加」为**跨风格保持**的能力；卫星纹理本身未消失，仅退出默认、并入天气风格包。故三场景所验的底层能力在重写后全部成立，**无 spec 子句掉出全部场景之外**。
- 裁决：**不降级、保留 ✅**。arch 的「不降级」提议采纳。
- 两条挂钩条件（须在 orch 应用 spec 重写时一并落地，否则 ✅ 语义失真）：
  1. **再归属留痕**：M1-05/06/14 现在实证的是**卫星/天气风格包路径**的昼夜/纹理表现，须在 testplan 行文或应用记录中注明其不覆盖「新默认矢量风格」，避免后续实例误读 ✅ 已覆盖矢量默认。此为「再归属」而非「改判据文字」，不触 §5.2 禁区（未下调判据强度、未使子句掉出场景）。
  2. **M2 必须新增矢量默认昼夜场景**：新默认风格改为矢量后，其昼夜/晨昏线呈现是零覆盖的新对外可见行为，须由 M2 FM-08 建实际 testplan 行（§4 已列「矢量默认昼夜」覆盖点，须落实为真实场景，不得停在占位）。否则 SPEC-3.2 的新默认矢量昼夜能力构成新的蒸发。
- 附带提示（非阻断，交 orch）：spec 重写后，现行 src/ 以卫星为默认的实现即与「默认矢量」不符——这是 M2 FM-08 的实现欠账（切换默认 + 卫星路径并入天气风格包），不构成对 M1 签核或 M1 ✅ 的动摇。

### 2.3 A4 — 合规调研（M5 收口）复验证据形式：**核对记录形式裁准，附依赖**

- 裁决：合规调研属「研究 + 结论」类交付，无自动化复现命令，正落 §5.3 第 4 条「确实写不出复现命令」的适用边界。其证据形式应为**独立 qa 出具的核对记录**（存 doc/evidence/，写明核对对象、逐条结论、遗留漏洞、署名），而非测试 log；**严禁挂无关回归 log 冒充**（REV-004 C2 已立此禁）。arch 的提议方向采纳。
- 依赖与边界（须交 orch 在 M5 前解决，否则 A4 无法机械落地）：
  1. **§5.1 判据张力**：§5.1 要求「该 M 的 testplan 场景全 ✅」，而 §5.2「没有测试 log 就没有 ✅」。合规若作为 M5 testplan 场景，其 ✅ 无测试 log 可挂——此即 **BUG-015**（OPEN，evidence.py bug 模式只吃测试 log，尚不支持核对记录型证据；已排 M2）。故 M5 合规收口须二选一：(a) BUG-015 的工具扩展落地并**延伸至场景级**（不止 bug 级）核对记录证据；或 (b) 合规收口建模为**独立签核子件**（类比 rev signoff），不进机械 ✅ 的 testplan 场景。
  2. 核对记录须满足 §5.3.4 内容底线，且严格限于确无复现命令者——凡能写出复现命令的合规子项（如 CoinGecko 署名是否实际渲染在详情卡）应走机械关单，不得借该口径绕过测试证据。

### 2.4 A6 — 里程碑由 4 扩为 5（M2–M6）：**M2–M6 成立**

- 裁决：M5（发布准备：PWA 升格 / 性能达标 / 合规收口 / 匿名遥测）与 M6（iOS 打包 / App Store 内购 / 原生 RSS）的拆分，判据独立性论证成立——PWA 离线壳闭环（承接 D9）、web RSS CORS 子集（D14）、性能达标（D10）、遥测（D18）自然聚为「零服务器发布准备」轨；Capacitor iOS 包 + IAP + 原生任意 feed（D14 原生全集）为另一轨，两轨判据（PWA service worker vs iOS 出包冒烟）互不覆盖，合并会使「PWA 升格为正式里程碑内容」失去独立完成判据。
- 未见 scope 越权：M2–M6 全部落在 D1–D21 范围内，属结构重排而非范围扩张。
